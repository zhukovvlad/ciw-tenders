# Спецификация: ревью/правки матчинга + запись «Статья СМР» + выгрузка .xlsx (SP3)

**Дата:** 2026-06-23
**Статус:** дизайн утверждён, готов к плану.
**Зависит от:** SP1 (парсер/персистентность сметы), SP2 (асинхронный матчинг + честный score).
**Предыдущие спеки:** [SP1](2026-06-22-estimate-hierarchical-parser-design.md), [SP2](2026-06-23-estimate-matching-sp2-design.md).

---

## Контекст

Третий под-проект модуля `estimates` — «живой документ + воркер». После SP1/SP2 смета
загружена, распарсена, узлы сматчены воркером: каждая строка-узел несёт **иммутабельный
AI-снимок** (`status`, `matched_*`, `score`, снимок `candidates`). SP3 даёт оператору
просмотреть результат, **поправить спорные строки вручную**, и **выгрузить `.xlsx`** с
заполненной колонкой `Статья СМР`.

Поток остаётся stateless по содержимому сметы: персистентен только снимок строк (как в
SP1/SP2). SP3 добавляет к строке **ось ревью** поверх неизменного AI-снимка.

Фронтенд уже несёт готовый по форме UX ревью (`pages/estimate/ReviewScreen`,
`ReviewRow`, редьюсер `lib/reviewState`) — но целиком на моках (`lib/mock/`). SP3
переключает его на реальный API.

---

## 1. Объём (Scope)

**Цель:** оператор просматривает матчинг, правит спорные строки, выгружает `.xlsx` с
колонкой `Статья СМР`.

**В объёме:**
1. **Отдача данных ревью** — расширить `EstimateRowOut`: `id`, `source_index`, снимок
   `candidates`, ось ревью (`review_status`, `final_*`, `reviewed_at`).
2. **Сохранение решений** — `PATCH .../rows/{row_id}/review`: подтвердить / выбрать
   кандидата / ручной подбор / забраковать. Пишет ось ревью, **не трогает** AI-снимок.
3. **Лексический поиск по справочнику** — `GET /articles/search` для ручного подбора
   (заменяет мок `searchArticles`).
4. **Экспорт `.xlsx`** — читать оригинал из MinIO, заполнить `Статья СМР` по правилу
   записи, отдать файл стримом.
5. **Переключить фронт ревью с моков** на реальный API.
6. **Правка SP2 (явная):** сузить matchable-фильтр и добавить CAS на запись матча, чтобы
   ре-триггер не стирал ручные правки.
7. **Staleness-sweep** (изолированно) — зависшие `running` после жёсткого краша воркера.

**Вне объёма:**
- Перекалибровка порога 0.90 (SP3 *сохраняет* данные для неё — провенанс правок, — но
  саму калибровку не делает).
- История/многопользовательское ревью одной строки (нет таблицы решений; одно терминальное
  состояние строки).
- rationale арбитра в UI (см. §7.1).
- Хранение сметы дольше текущего snapshot-контракта.
- `Статья СМР` на строках-позициях (листьях) — их SP1 не сохранял (см. §5, край D).
- Beat/глобальный фоновый sweep (только per-estimate на ре-триггере, §6).

## 2. Модель данных (две оси)

Каждая строка-узел `estimate_rows` несёт **две независимые оси**.

### Ось AI (иммутабельна — из SP2, SP3 её не меняет)
`status` (`pending`/`confident`/`needs_review`/`no_match`/`error`), `matched_article_id`,
`matched_code`, `matched_name`, `score`, `candidates` (JSONB-снимок `[{id,code,name,score}]`).

### Ось ревью (новое — добавляется к строке)

| Колонка | Тип | Смысл |
|---|---|---|
| `review_status` | text NOT NULL default `'unreviewed'` | `unreviewed` → `confirmed` \| `overridden` \| `rejected` |
| `final_article_id` | int NULL | id выбранной статьи, **заморожен** (plain int, без FK — как `matched_article_id`) |
| `final_code` | text NULL | заморожен |
| `final_name` | text NULL | заморожен |
| `reviewed_at` | timestamptz NULL | отметка решения; отличает `rejected` (NULL final + статус) от `unreviewed` |

### Семантика осей (провенанс для будущей калибровки — восстановим без доп. таблицы)
- `confirmed` → `final_article_id == matched_article_id` (человек согласился с
  **рекомендацией AI** `matched_*` — это retrieval-top-1 для `confident` и выбор арбитра для
  `needs_review`). На confirm пишем `final_* = matched_*` **явно** (единообразие экспорта).
- `overridden` + `final_article_id` ∈ id снимка кандидатов → выбрал другого кандидата.
- `overridden` + `final_article_id` ∉ кандидатов → ручной подбор из справочника.
- `rejected` → `final_* = NULL`, `review_status='rejected'` (явно «статьи нет»).

**Ограничение провенанса (связано с краем A, §3.4):** калибровочная корзина
`overridden ∉ candidates` смешивает «честный ручной подбор» и «pick по устаревшему
кандидату» (§3.4) — они там неразличимы. Сейчас это **принятое загрязнение**. Если
калибровка потом потребует чистого сигнала «ручной vs кандидат», единственный способ убрать
загрязнение — записывать источник pick отдельным полем провенанса (`pick_source`); в SP3
такого поля **нет** (YAGNI), загрязнение принимаем сознательно.

### Заморозка `final_*` (источник зависит от способа выбора — край B)
`final_*` морозятся **в момент решения**, не из live-данных в момент чтения/экспорта:
- **Кандидат из снимка** (`article_id` ∈ `candidates`) → `final_*` из записи `candidates`
  (а не из live `template_articles`) — если справочник потом изменят, решение не «поедет».
- **Ручной подбор / поиск** (`article_id` ∉ `candidates`) → источника в снимке нет по
  определению; морозим `code`/`name` из `template_articles` **на момент PATCH**. Принцип
  «заморожено в момент решения» соблюдён; просто источник заморозки — справочник, а не
  снимок. Если статью потом удалят/переименуют — `final_*` уже скопированы.

### Стык с ре-триггером SP2 (matchable-фильтр)
matchable = `status ∈ {pending, error, no_match}` **И** `review_status = 'unreviewed'`.
Ручная правка иммутабельнее AI-снимка; переопределённый `no_match` больше не перематчивается.

### Миграция / ORM
Одна alembic-ревизия: +4 колонки в `estimate_rows` (`review_status` NOT NULL default
`'unreviewed'`, `final_article_id` int NULL, `final_code` text NULL, `final_name` text NULL,
`reviewed_at` timestamptz NULL). ORM-модель `EstimateRowModel` синхронно. Опционально —
композитный индекс `(estimate_id, status, review_status)` под matchable-фильтр. `gin_trgm`
под поиск — отдельный enhancement (§4).

## 3. API: чтение ревью + правка решения

### 3.1 Расширение `GET /estimates/{id}` (`EstimateRowOut`)
Добавить поля: `id` (PK строки — адресует правку), `candidates`
(`[{id,code,name,score}]`), `review_status`, `final_article_id`, `final_code`,
`final_name`, `reviewed_at`. **rationale не добавляем** (§7.1). `source_index` наружу **не
отдаём** — экспорт серверный, фронту он не нужен (лишняя внутренняя деталь).

Требует, чтобы доменный `StoredEstimateRow` и repo `get()` несли снимок кандидатов и
ось ревью (сейчас `get()` их не селектит).

### 3.2 `PATCH /estimates/{estimate_id}/rows/{row_id}/review`
Body (DTO): `{ "action": "confirm" | "pick" | "reject", "article_id": int | null }`.

- **Владение** как в `get` (через `requester_id` + `is_admin`): чужая/несуществующая → 404.
- **Разводка по статусу строки (край F):**
  - `pending` → **409** («строка ещё не сматчена» — воркер не дошёл).
  - `error` → confirm запрещён (**422**), доступны pick / reject.
  - `confident` / `needs_review` → доступны confirm / pick / reject.
  - `no_match` → confirm запрещён (**422** — нет top-1), доступны pick / reject.
- **confirm** — только при наличии `matched_article_id` (`confident`/`needs_review`). →
  `review_status='confirmed'`, `final_* = matched_*`, `reviewed_at = now`.
- **pick** (`article_id` обязателен):
  - `article_id` ∈ id снимка `candidates` → морозим `final_*` из кандидата.
  - иначе → ищем в `template_articles` по id → морозим `final_*` из справочника.
  - нет ни там, ни там → **422** («статья не найдена»).
  - `review_status = 'confirmed' if article_id == matched_article_id else 'overridden'`.
- **reject** — любая сматченная строка: `review_status='rejected'`, `final_* = NULL`,
  `reviewed_at = now`.
- Возвращает обновлённую `EstimateRowOut`.
- **Правка авторитетна** — пишет ось ревью без условия на текущий `review_status`
  (оператор может передумать и переписать своё решение).

### 3.3 Синхронизация правка ↔ ре-триггер — CAS на стороне писателя
Гонку закрывает **CAS на записи AI-снимка**, а не лок на правке (правку не блокируем —
оператор ревьюит, пока воркер доматчивает другие строки):
- `fetch_matchable_nodes`: + `AND review_status = 'unreviewed'`.
- `save_node_match`: `UPDATE ... WHERE id = :node_id AND review_status = 'unreviewed'`.
  Если человек тронул строку в окне read→write — CAS даёт 0 строк → AI-снимок не затирает
  решение. Снимок кандидатов и `final_*` остаются взаимно-консистентными (оба «старые»,
  ровно то, что видел оператор).

Это явный touch SP2-репозитория (`estimate_repository.py`).

### 3.4 Известный край: pick по устаревшему кандидату (край A)
Сценарий: ре-триггер прошёл gate и **успел** перематчить строку (`no_match→confident`,
`save_node_match` прошёл, т.к. строка была ещё `unreviewed`) и закоммитил; **сразу после**
оператор шлёт `pick article_id=X`, где X — кандидат из *старого* снимка, которого новый
матч уже не содержит. Ветка pick: `X ∉ новых candidates` → уходит в «ищем в
`template_articles` → морозим из справочника». То есть **pick по устаревшему кандидату
молча деградирует в ручной-подбор**, а не падает.

- **Последствие:** экспорт корректен (`final_*` заморожены, статья валидна). Но провенанс
  искажается: `overridden`-ручной vs `overridden`-кандидат — это сигнал для калибровки.
- **Разрешение (на текущем масштабе):** фронт **перечитывает** строку (`getEstimate`)
  после ре-триггера и перерисовывает ревью перед сабмитом правки; вероятность окна мала.
  Жёстче (если понадобится) — PATCH принимает `reviewed_at`/версию снимка и на рассинхроне
  отвечает **409** «строка перематчена, обнови». В SP3 берём «фронт перечитывает»; край
  назван явно как известная деградация (ручной-подбор, не ошибка), а не молчаливый баг.

## 4. Поиск по справочнику (лексический)

`GET /articles/search?q=...&limit=20` — любой аутентифицированный пользователь.

- **Серверный мин-длина `q`:** `len(q.strip()) >= 2` — иначе **400** (или пустой список).
  Без guard пустой/односимвольный `q` даёт `ILIKE '%%'` → полный seq-scan, обрезанный
  `limit`. Фронт дебаунсит, но не-фронтовый вызов иначе выгребет всё.
- **Лексический**, не семантический: `WHERE code ILIKE '%q%' OR name ILIKE '%q%'`,
  `ORDER BY code`, `LIMIT`. Возвращает `[{id, code, name}]` (без `score` — это не сходство).
- Без AI/эмбеда: предсказуемый результат по коду/названию; семантику оператор уже получил
  от AI. Для ручного подбора нужен детерминизм, а не «похоже по смыслу».
- **`search()` НЕ фильтрует по `embedding`** (край B): оператор должен видеть **все** статьи
  справочника, включая ещё не заэмбеженные (`embedding IS NULL`). Для ручного подбора вектор
  не нужен — морозим `code`/`name`. (Это отличие от поискового матчинга SP2, который
  исключает `embedding IS NULL`.)
- Новый метод `search(q, limit)` в `ArticleRepository` (порт + адаптер) + роут в
  `articles.py`.
- Опционально: `gin_trgm_ops` индекс по `name` (миграция), если ILIKE с ведущим `%`
  окажется медленным на большом справочнике. На текущем размере seq-scan приемлем — индекс
  как enhancement, не блокер.

Фронт: дебаунс + мин-длина запроса (не эмбедим/не дёргаем на каждую букву — здесь даже без
эмбеда лишние запросы ни к чему).

## 5. Экспорт `.xlsx`

`GET /estimates/{id}/export[?strict=true]` → стрим `.xlsx`.

### Поток
1. Владение (как `get`) → чужая/нет → 404.
2. Сервис берёт `original_object_key` (живёт внутри репо, наружу не отдаётся) →
   `ObjectStorage.get(key)` → байты.
3. `openpyxl.load_workbook(BytesIO(bytes))`, активный/нужный лист.
4. Найти колонку `Статья СМР` (правило ниже); нет → создать в `max_column + 1`.
5. Проход **только по сохранённым узлам** (строки БД): записать `Статья СМР` по правилу.
6. `wb.save(buf)` → `StreamingResponse(BytesIO)`.

### Правило поиска колонки (край C — детерминизм)
- Заголовки гарантированно в **строке 1** — это инвариант SP1 (файл уже прошёл
  fail-closed-парсер на ingest; невалидная структура не попала бы в MinIO). Опираемся на
  него явно.
- Матч заголовка по **нормализованному равенству** `strip().casefold()` (а не `==`): ловит
  `«Статья СМР »` с хвостовым пробелом, регистр и т.п.
- Несколько колонок с похожим именем → берём **первую слева**.
- Колонки нет вообще → создаём в `max_column + 1`, пишем заголовок в строку 1. Добавление
  **справа** не сдвигает `source_index`-маппинг (физические строки данных не двигаются).

### Правило записи (по строке-узлу)
`physical_row = source_index + 2`. Экспорт грузит **оригинал** через openpyxl (1-based,
строка 1 = заголовок), а `source_index` пришёл из `df.iterrows()` SP1-парсера (0-based
RangeIndex, pandas съедает заголовок). Маппинг `+2` держится, **пока** SP1-чтение не меняет
инварианты: заголовок в строке 1, никаких `skiprows`/`dropna`/`reset_index`, строки не
выбрасываются. **Связка явная:** если SP1 однажды изменит `read_excel`, экспорт начнёт молча
писать не в те ячейки — поэтому **экспорт и SP1-чтение меняются в одном такте**, и golden
раунд-трип-тест (ниже) ловит рассинхрон офсета.
`ws.cell(row=physical_row, column=col).value = <значение>`:

| Состояние строки | Значение в `Статья СМР` |
|---|---|
| `review_status ∈ {confirmed, overridden}` | `final_code` |
| `review_status = rejected` | пусто |
| `unreviewed` + `confident` | `matched_code` (авто-принято) |
| `unreviewed` + `needs_review` / `no_match` / `error` / `pending` | пусто |

> **Амендмент 2026-07-02** (ревью фонда, см.
> [2026-06-30-decision-fund-design.md §12.1](2026-06-30-decision-fund-design.md)):
> (1) добавлена строка матрицы `unreviewed + matched_fund` → статья из снимка (`matched_*`) —
> авто-принято фондом, как `confident`; (2) формат значения ячейки во всех непустых ветках —
> **`(код) Название`** (`final_code`+`final_name` либо `matched_code`+`matched_name`); без имени —
> голый код, без кода — пусто.

**Решение по `unreviewed + needs_review` → пусто (сознательный продуктовый выбор):**
`needs_review` по определению — «AI не уверен» (самая рискованная авто-подстановка). Запекать
её в `.xlsx` как `matched_code` неотличимо от подтверждённой = молча выдать негарантированную
догадку за финальный код и подорвать смысл шага ревью (флаг «непросмотрено» в файл не уезжает).
Поэтому `needs_review` без ревью экспортируется **пусто** — хочешь код, подтверди/выбери.
`confident` остаётся авто-заполненным осознанно: высокий score — это и есть замысел
авто-принятия. (Альтернатива «заполнять `needs_review` догадкой» отвергнута как accepted-risk,
которого мы не берём.)

### Строки-позиции (листья) не трогаются (край D)
Экспорт пишет **только в строки-узлы** (у позиций-листьев нет `source_index`/строки БД —
SP1 их не сохранял). В выгрузке ячейка `Статья СМР` у строк-позиций остаётся **как в
оригинале** (обычно пусто). Это ожидаемое поведение; явно проговорено, чтобы на демо не было
сюрприза «почему половина строк пустая». `Статья СМР` на позициях — отдельный разговор (за
рамками SP3; упирается в то, что снимок позиций отброшен ещё в SP1).

### Гейт и сводка (частичный экспорт по умолчанию)
- **Дефолт** — частичный экспорт всегда (кнопка «Выгрузить» достижима на больших сметах).
- **`?strict=true`** — если есть `needs_review`/`no_match` с `review_status='unreviewed'` →
  **409** со счётчиком непросмотренных (серверный гейт для желающих жёстко).
- **Сводку** («N строк с AI-догадками без ревью, M пусто») фронт считает из уже загруженных
  строк и показывает у кнопки — не тащим в бинарный ответ.

### Стрим, память, ошибки (край E)
- `StreamingResponse` с `BytesIO` после `wb.save(buf)`.
  `Content-Disposition: attachment; filename="<имя>_matched.xlsx"`,
  `media_type=application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`.
- Полная загрузка workbook в память приемлема для смет масштаба golden (2762 строки). Грузим
  нужный лист.
- `ObjectStorage.get` упал (MinIO недоступен) → **503** (как enqueue/ingest в SP2/SP1), не
  500.

## 6. Staleness-sweep (изолированно)

Закрывает долг из TECH_DEBT: смета «залипла в `running` навсегда» после жёсткого краша
воркера (SIGKILL/OOM — Python `except` не отработал; Postgres лок отпустил, но строку никто
не перезаписал).

**Инициатор — роут ре-триггера** `POST /estimates/{id}/match` (естественное действие
«продвинь смету»; заодно чинит вводящий в заблуждение `detail` из TECH_DEBT). Без
beat/глобального скана (вне скоупа).

**Логика (перед enqueue):**
- Если `status = 'running'` И `now - updated_at > task_time_limit_s` (660s):
  живой прогон **всегда свеж** из-за heartbeat'а SP2 (`touch` каждый чанк; макс. интервал
  между бампами = hard-лимит по определению), мёртвый — `updated_at` застыл.
  - `try_advisory_lock(estimate_id)` как **арбитр живости**: взялся → прежний держатель
    мёртв → `running → pending`, отпустить лок; не взялся → задача жива (медленная) → no-op.
- Затем enqueue как сейчас. `detail` теперь честный: «перезапущено после сбоя» / «уже
  выполняется» / «поставлено в очередь».

Порог `> task_time_limit_s` безопасно больше любого здорового интервала heartbeat'а;
`try_advisory_lock` гарантирует, что sweep не подерётся с живым воркером.

## 7. Решённые открытые вопросы

### 7.1 rationale арбитра — выкинуть из UI в SP3
SP2 не хранит «почему» (арбитр возвращает только индекс). Хранение = доработка SP2-промпта
(вернуть текст) + колонка + миграция — расширение уже залоченного SP2 ради UI-украшения.
Оператору на ревью важнее **кандидаты со score** (есть в снимке) и возможность override.
Если позже понадобится для калибровки — добавится отдельным проходом. Сейчас: показывать
`candidates` со score, «Почему» не тащить (удалить из `ReviewRow`).

### 7.2 Гейт экспорта — частичный по умолчанию + сводка
Жёсткий гейт «все спорные тронуты» сделал бы «Выгрузить» недостижимой на больших сметах.
Дефолт — частичный экспорт + клиентская сводка; `?strict=true` для серверного гейта (409).
См. §5.

## 8. Затронутые компоненты (карта реализации)

  заморозки), сценарий экспорта (сборка workbook), staleness-sweep helper.
- `infrastructure/db/estimate_repository.py` — миграционные правки запросов (matchable CAS),
  `get()` селектит candidates + ось ревью, запись решения.

- `infrastructure/db/models.py` — +4 колонки `EstimateRowModel`.
- `infrastructure/` экспорт — openpyxl-сборка из байтов MinIO (адаптер/хелпер; внешний
  ввод-вывод — здесь, не в services).
- `api/routes/estimates.py` — `PATCH .../rows/{id}/review`, `GET .../export`, sweep в
  `retrigger_match`.
- `api/routes/articles.py` — `GET /articles/search`.
- `api/schemas.py` — `EstimateRowOut` (+поля), `ReviewDecisionIn`, `ArticleSearchOut`.
- `alembic/versions/` — ревизия +4 колонки (+опц. индексы).

**Frontend:**
- `lib/api/estimates.ts` (новый) — `getEstimate` (с кандидатами/осью ревью), `patchRowReview`,
  `exportEstimate` (download), и `searchArticles` в `lib/api/articles.ts`.
- `pages/estimate/EstimateFlow`, `ReviewScreen`, `ReviewRow` — с `lib/mock` на `lib/api`;
  маппинг DTO→`MatchRow` (`row_number`←`id`, `source_name`←`name`,
  `candidates[].article_code`←`code`); убрать `rationale`; дебаунс+мин-длина на поиске.
- `lib/reviewState` — действия (`confirmArbiter`/`pickCandidate`/`manualPick`/
  `confirmNoMatch`) маппятся на `PATCH` action (`confirm`/`pick`/`pick`/`reject`).

**Тесты:** юнит на фейках портов (без сети/БД/AI) — правка решения (все ветки action ×
статус, CAS-консистентность, заморозка снимок-vs-справочник), поиск (ILIKE, не фильтрует
embedding, мин-длина `q`), экспорт (правило записи по матрице вкл. `needs_review→пусто`,
только узлы, поиск заголовка нормализацией, 503 на MinIO), sweep (порог + арбитраж лока).
**Golden раунд-трип экспорта** (тот же страж офсета, что в SP1): на golden-файле известный
узел (напр. `1.1.5 → source_index 33`) с заданным `final_code` должен записаться ровно в
физ.строку `35` оригинала — ловит рассинхрон `+2`/SP1-чтения. Фронт: vitest на ревью с
реальным API-слоем (замоканный fetch).

## 9. Порядок реализации (для плана)

Строго по слоям и зависимостям; staleness-sweep — независимый кусок:
1. **Миграция + ORM + домен** (ось ревью, снимок в `StoredEstimateRow`) — фундамент.
2. **SP2-стык**: сужение `fetch_matchable_nodes` + CAS `save_node_match` (защита правок до
   того, как появятся сами правки).
3. **Чтение ревью**: `get()` селектит candidates/ось, `EstimateRowOut` +поля.
4. **Правка решения**: сервис + `PATCH` (все ветки A/B/F).
5. **Поиск справочника**: repo `search()` + `GET /articles/search`.
6. **Экспорт**: storage `get`, openpyxl-сборка, `GET .../export` (C/D/E).
7. **Staleness-sweep**: helper + интеграция в `retrigger_match`.
8. **Frontend**: переключение с моков, маппинг, дебаунс, удаление rationale.

Каждый шаг: `uv run ruff check .` + тесты зелёные; фронт — `npm run typecheck` + vitest.
