## 1. Проблема

Крошка узла сметы (`embedding_input`) — это «Имя(предок). … . Имя(узел)». Сегодня предки
восстанавливаются **усечением точечного кода** (`6.4.1` → предки `6`, `6.4`) с подстановкой имени
из словаря `код → имя` **первого вхождения**
([estimate_parser.py:67-84](../../../backend/app/services/estimate_parser.py#L67),
[estimate_matching_service.py:154-158](../../../backend/app/services/estimate_matching_service.py#L154)).
Иерархия сметы целиком выводится из кода — а коды в реальных файлах часто кривые. Три измеренных
режима отказа (golden «Смета — образец», 809 узлов-кодов):

| Режим | Пример | Что даёт код сегодня | Истинный предок |
|---|---|---|---|
| **Коллизия (дубль кода)** | `6.4.1` в Этапе 1 и Этапе 2 | имя первого вхождения `6.4` (Этап 1) для обоих | разные этапы |
| **Родитель ниже** | `10.2.1.1` (si=1533), его `10.2.1` впервые на si=1857 | «Фасадные осветительные приборы **2 Этап Офис**» (снизу, чужой этап) ❌ | «Наружное освещение **1 Этап ЖК**» |
| **Родителя нет** | газон `11.3.1.1.1`, кода `11.3.1.1` в смете нет вовсе | предки `None` (потеряны) ❌ | «Покрытие по грунту» |

Баг **тихий** (строка молча уезжает к неверной статье). Бьёт по всем многоэтапным сметам.

**Корень общий для всех трёх:** структура выводится из **кода** (теряет позицию, ломается на
дублях/пропусках), а не из фактического положения строки в документе.

### Сигнал истины: `outlineLevel` присутствует и богат

В реальных сметах есть **нативная группировка строк Excel** (`row_dimensions[r].outline_level`):
на golden — уровни 0–6, тысячи строк. Это независимый позиционный сигнал вложенности, который
`pandas.read_excel` сейчас выбрасывает.

**Важно — outline ≠ глубина-кода на 14% узлов (измерено):** `outline_level+1 == число_сегментов`
совпадает на 694 узлах и **расходится на 115 (14.2%)**. Расхождения — НЕ ошибки: это ровно
многоэтапные места, где outline корректно добавляет уровень этапа, пропущенный кодом (первые —
фасадные `6.2.x`: outline-глубина 3 vs код-глубина 2, потому что строка-этап «1 Этап ЖК» — реальный
уровень группировки, а код её перескакивает). То есть «outline совпадает с кодом» — свойство только
**корректных** строк; на коллизионных они расходятся, и outline там прав. Это прямое следствие для §4
(объём `outline_code_mismatch`).

### Замер альтернатив (golden, 809 узлов)

| Сравнение | Расходятся узлов | Смысл |
|---|---|---|
| сегодня (код-усечение, первое вхождение) ↔ outline-стек | **68** | насколько нынешняя логика «врёт» против идеала |
| **fallback (стек по глубине-кода, без outline)** ↔ outline-стек | **27** | что теряем без outline |
| сегодня ↔ fallback | **50** | fallback **сам** уже лучше сегодняшнего |

Оба «битых» кейса (`10.2.1.x`, газон) чинятся **и outline-стеком, и fallback'ом**. Бо́льшая часть
из 27 расхождений — спор «непосредственный предок = ORG-этап vs глава», который **смывается
орг-стрипом** (этапы и так выбрасываются из крошки), т.е. реальная разница в крошке ещё меньше.

## 2. Решение: единый позиционный стек по глубине

Идём по строкам **в порядке документа**, держим стек «глубина → последний открытый узел этой
глубины». Родитель узла = вершина стека на `глубина − 1`; полная цепочка предков = вершины на всех
уровнях `< глубина`.

**Глубина узла:**
- если в файле есть группировка (`max(outline_level) ≥ 1`) → `depth = outline_level + 1`;
- иначе (плоский файл) → `depth = число сегментов кода` (`11.3.1.1.1` → 5).

**Почему это строго сильнее кода и поглощает Кейс D:**
- стек содержит **только строки выше** → «контекст снизу» (`10.2.1.x`) становится **структурно
  невозможным**, а не «лечится»;
- имена предков больше **не зависят от кода** — только от позиции → коллизии дублей разводятся
  естественно;
- пропущенный код-родитель (газон) не теряет цепочку — родитель берётся из ближайшего открытого
  уровня выше.

**Отличие от поглощённого Кейса D:** там был nearest-preceding **по коду** (флаг `is_fallback`,
тянул контекст снизу на откатных узлах). Здесь стек **по глубине** — тянуть снизу нельзя в принципе,
флаг `is_fallback` не нужен, монотонность индексов безусловна.

## 3. Архитектура

Направление зависимостей и слои — без изменений (`api → services → domain ← infrastructure`).

### §3.1 Домен — чистые функции

В [domain/classification.py](../../../backend/app/domain/classification.py):

```python
def resolve_ancestor_indices(depths: Sequence[int]) -> list[list[int]]:
    """depths — глубина каждого узла В ПОРЯДКЕ документа (1-based, ≥1).
    Для позиции i возвращает индексы предков root→parent: для каждого уровня
    d в 1..depths[i]-1 — последний индекс j < i с depths[j] == d, ещё «открытый»
    (не закрытый более мелким узлом между j и i). Пропущенный уровень (нет
    открытого узла глубины d) выпадает из цепочки.

    Инвариант: возвращаемые индексы строго возрастают и все < i (стек — только
    предшествующие строки). Тестируется property-тестом; флаг is_fallback не нужен."""

def node_depth(outline_level: int, segment_count: int, file_has_outline: bool) -> int:
    """Глубина узла: outline_level+1 если в файле есть группировка, иначе segment_count."""
```

- Принимают **только числа** (глубины) → чистые, без ORM/имён/классов; тривиально тестируются.
- Маппинг `индекс → (имя, класс)` остаётся в сервисах — там, где известны имена и классификация.

### §3.2 Чтение `outline_level` в парсере

[estimate_parser.py](../../../backend/app/services/estimate_parser.py): pandas остаётся для значений
(сохраняет проверенный `dtype={"№ раздела": str}` против `1.10 → 1.1`). **Добавляется тонкий проход
openpyxl** по тому же `content` (read-only), извлекающий `outline_level` по физ.строкам, выровненный
по `source_index` (физ.строка = `source_index + 2`). `file_has_outline = max(levels) ≥ 1`.

> **🔴 Критический риск — рассинхрон pandas↔openpyxl по офсету `+2`.** Этот патч впервые **сшивает
> два прохода** по позиции. Если openpyxl-проход и pandas-фрейм разъедутся хоть на строку (пустая
> строка в середине, иначе схлопнутая pandas-ом), получим не порченую коллизию, а **сдвиг глубины у
> ВСЕХ узлов ниже точки рассинхрона** — новый тихий баг, хуже чинимого. На golden он не выстрелит
> (измерено: 0 пустых строк, `pandas[si]` vs `openpyxl[si+2]` — 0 расхождений), поэтому round-trip на
> golden его НЕ ловит. «Инвариант +2 у экспортёра» это не закрывает: что экспортёр *пишет* по +2 —
> не доказывает, что openpyxl-*чтение* выровнено с тем, как pandas роняет строки.
>
> **Закрываем не доверием, а жёстко:**
> 1. **Ассерт на каждый узел при парсинге:** код, прочитанный openpyxl по `source_index+2`, обязан
>    байт-в-байт совпасть с кодом из pandas по `source_index`. Не совпало → падаем громко (`ValueError`
>    до записи куда-либо, как для отсутствующих колонок), а не молча эмбеддим кривую глубину.
> 2. **Отдельный тест** с пустой/съехавшей строкой, которая *должна* была бы рассинхронить два прохода
>    → проверяет, что ассерт ловит. Round-trip на чистом синтетике это НЕ покрывает.
>
> Рассмотрена и отклонена альтернатива «один источник — только openpyxl» (убрала бы риск в принципе):
> крупнее, теряет проверенное pandas-поведение `dtype=str`; ассерт дешевле и сохраняет инвариант.

### §3.3 Три точки резолва переключаются на стек

Все мапят `resolve_ancestor_indices(depths)` → `(имя, класс)` по индексу:

1. **`_classify_nodes`** ([estimate_matching_service.py:148-175](../../../backend/app/services/estimate_matching_service.py#L148))
   — **авторитетный путь** крошки (чинит эмбеддинг/матчинг). Заменяет `_ancestor_codes` + словарь
   `name_by_code`/`repr_id_by_code` на позиционный резолв.
2. **`_ancestor_names`** ([estimate_matching_service.py:187](../../../backend/app/services/estimate_matching_service.py#L187))
   — контекст предков для LLM-классификатора.
3. **`estimate_parser`** — крошка на парсинге (consistency; её перезатирает классификация, но два
   расходящихся способа строить дерево держать нельзя).

Глубины (`depths`) для сервисных точек прокидываются из узлов (`EstimateNode` получает `depth` уже
вычисленной парсером — поле `depth` существует; теперь оно из `node_depth`, а не из `len(segments)`).

### §3.4 `is_leaf` мигрирует на позиционный сигнал

`is_leaf` ([estimate_matching_service.py:160](../../../backend/app/services/estimate_matching_service.py#L160))
кормит leaf-rescue (`is_excluded`): отличает «орг-похожий лист = реально работа» (оставить) от
«орг-заголовок с детьми» (исключить). Сейчас код-based (`n.code not in parent_codes`) → болеет тем же
багом. Позиционный стек даёт детность бесплатно: **узел — лист ⟺ следующий узел не имеет бо́льшую
глубину** (никто не открывается под ним до закрытия его уровня). Вводится доменный помощник
`leaf_flags(depths) -> list[bool]` (или вычисление из той же проходки стека) и заменяет `parent_codes`.
`classify_lexical` / LLM-классификатор / `is_excluded`-логика — не трогаются.

## 4. Детекция аномалий (контроль при загрузке)

На парсинге собираем **типизированные** аномалии (не плоские строки). Доменный тип:

```python
@dataclass(frozen=True)
class StructuralAnomaly:
    kind: str            # см. таблицу
    source_index: int
    code: str
    name: str
    detail: str          # человекочитаемое («родитель '10.2.1' НИЖЕ: si=1857 …»)
```

**Построчные (редкие, действенные) — в таблицу отчёта:**

| kind | условие | смысл |
|---|---|---|
| `duplicate_code` | код встречается ≥2× | коллизия (Кейс D) |
| `parent_below` | код-родитель впервые встречается ниже узла | нарушен порядок |
| `parent_missing` | код-родителя нет в смете вовсе | пропущен уровень |
| `depth_jump` | глубина растёт больше чем на 1 | возможен неверный предок из стека |

**Агрегат (НЕ построчно) — `outline_code_mismatch`.** Измерено: срабатывает на **115 из 809 (14%)**
узлов на golden — это **базовое состояние** многоэтапной сметы (ровно те места, где outline
правильно взят вместо кода, §1), а не редкая ошибка. Высыпать его построчно — утопить оператора в
14% строк и спрятать реально редкие `duplicate_code`/`parent_below`/`parent_missing`/`depth_jump`.
Поэтому он отдаётся **одной агрегатной строкой**: «в N строках вложенность взята из группировки Excel,
а не из кода» — как справка, не как список к разбору.

**Где течёт:** `ParsedEstimate.anomalies: list[StructuralAnomaly]` (построчные) + агрегатный счётчик
`outline_overrides: int` → `EstimateService` → DTO в [schemas.py](../../../backend/app/api/schemas.py)
→ ответ `POST /estimates` ([estimates.py:103](../../../backend/app/api/routes/estimates.py#L103)).
Существующие `warnings` остаются (общие предупреждения парсера).

**Фронт:** сворачиваемый блок «Структура сметы: N замечаний» с таблицей построчных аномалий
(kind / код / имя / detail) + строка-агрегат про outline. По аномальным строкам показываем
**«по коду = X → взяли по структуре = Y»** (предок по коду-усечению vs по позиционному стеку) —
оператор руками не правит, но может перепроверить спорные.

## 5. Экспорт: комментарии + опц. ремонт нумерации — ⏭️ ОТДЕЛЬНЫЙ ПОСЛЕДУЮЩИЙ ПАТЧ

> **Почему отрезано (характер отказа, не объём).** Ремонт **перезаписывает авторитетный `№ раздела`,
> который потребляет другая автоматическая система**. Его корректность держится на позиционном дереве —
> а дерево в этом патче только вводится и подтверждается пост-замером (§6) **внутри этого же патча**.
> Отгружать наружу канонические коды, опираясь на ещё не подтверждённое дерево, нельзя: оставшийся
> `depth_jump`-промах уедет не в крошку (один деградировавший матч, восстановимо), а в **ключ чужой
> системы** — тихо и куда тяжелее ловится. Поэтому: отгружаем §2–§4, снимаем пост-замер, убеждаемся
> что дерево верное — и **на доказанном дереве** делаем ремонт отдельной задачей. Разрез почти
> бесплатный: **доменная `canonical_codes` остаётся в этом патче** (с юнитами, §6), уезжает только
> wiring экспорта (галочка, перезапись столбца, round-trip).

Ниже — дизайн отложенной части (фиксируем сейчас, реализуем после подтверждения дерева).

[estimate_export_service.py](../../../backend/app/services/estimate_export_service.py) пишет по
физ.строке `source_index + 2` (инвариант сохраняется).

- **Доп. столбец «Замечания по структуре»** — пер-строчные комментарии по аномалиям этой строки
  (тип + detail). На неаномальных строках пусто.
- **Галочка «Исправить нумерацию»** (параметр экспорта, по умолчанию ВЫКЛ). Когда включена:
  - в столбец **`№ раздела` пишутся канонические коды**, восстановленные из позиционного дерева
    (**полная канонизация**: код = путь порядковых номеров среди братьев в восстановленном дереве —
    `1`, `1.1`, `1.1.1`, …; обход в порядке документа). Перезапись исходного столбца — намеренно:
    смета далее уходит в **другую автоматическую систему**, которой нужны чистые коды в штатном
    столбце;
  - **оригинальный код** этой строки выносится в столбец «Замечания по структуре»
    (`ориг. код: 10.2.1.1`) — трассируемость.
- Когда галочка ВЫКЛ — `№ раздела` не трогаем, пишем только столбец замечаний.

Канонизация — чистая доменная функция `canonical_codes(depths) -> list[str]` (из той же позиционной
проходки), тестируется без БД.

## 6. Тесты (TDD: red → green)

**Юнит (чистые, без БД/AI):**
- `resolve_ancestor_indices`:
  - коллизия `6.4.1` ЖК/БЦ → **разные** цепочки предков;
  - «родитель ниже» (`10.2.1.x`) → берётся предшествующий узел, не «2 Этап Офис»;
  - «родителя нет» (газон `11.3.1.1.x`) → ближайший открытый уровень («Покрытие по грунту»);
  - property-монотонность: индексы строго возрастают и `< i`.
- `node_depth`: outline есть → из outline; плоский → из сегментов.
- `leaf_flags`: лист vs узел-с-детьми на позиционном дереве (включая коллизию кодов).
- `canonical_codes`: дерево → `1/1.1/1.1.1`; устойчиво к битым исходным кодам. (Функция в домене
  уже в этом патче; её экспортный потребитель — в отложенном §5.)
- Детектор аномалий: каждый построчный `kind` срабатывает на сконструированном входе; агрегат
  `outline_overrides` считается верно.

**Сервис-тест `_classify_nodes`:** на коллизирующих кодах `embedding_input` двух строк выходит
**разным** (сейчас — байт-в-байт одинаков). Ловит интеграцию в авторитетный слой.

**🔴 Тест рассинхрона pandas↔openpyxl (обязательный, §3.2):** синтетический `.xlsx`, где пустая/
съехавшая строка *должна* была бы сдвинуть один проход относительно другого → ассерт выравнивания
падает громко. Round-trip на чистом синтетике этого НЕ покрывает — нужен именно «грязный» вход.

**Round-trip парсер→outline (этот патч):** синтетический `.xlsx` с группировкой строк — `outline_level`
прочитан и выровнен; `file_has_outline` корректен; на плоском файле — fallback на сегменты.
*(Экспортный round-trip — галочка/перезапись/оригинал-в-замечания — в отложенном §5.)*

**Пост-замер (разово, не в CI; OpenRouter живой):**
- **переэмбеддить узлы** golden-сметы (крошки изменятся), `eval_matching` до/после → дельта
  **top-1 / top-3**; справочник переэмбеддивать **не** нужно (его крошки не трогаем);
- **именованный срез:** узлы `10.2.1.x` (родитель ниже) и газон `11.3.1.1.x` — флипает ли их top-1;
- метрика числа узлов с изменённым предком (ожидаем ≈ **68** на golden);
- счётчик `outline_overrides` (ожидаем ≈ **115**) — не метрика качества, а калибровка UX отчёта под
  реальный объём (чтобы спроектировать его под 14%, а не под горстку, см. §4).

## 7. Границы скоупа

**Не трогаем:** org-стрип, порог 0.90, LLM-арбитр, каталог, схему БД, экспортный офсет `+2`,
`classify_lexical`/LLM-классификатор, `is_excluded`-правило.

**Не делаем сейчас:** ремонт нумерации в экспорте (§5 — отдельный патч на подтверждённом дереве);
ранжирование/крошку справочника (Кейсы A–C — отдельный разговор); иерархический матчинг (роллапы N→1).

**Риск-заметки:**
- **🔴 Рассинхрон pandas↔openpyxl по `+2`** — самое опасное место патча (впервые сшиваем два прохода).
  Закрыт жёстким пер-узловым ассертом + «грязным» тестом, НЕ доверием к офсету (§3.2, §6).
- `depth_jump` (скачок глубины) — единственный остаточный режим, где стек может взять неверного
  предка (уровни между пропущены, на стеке — стейл от прошлой ветки). Детектируется как аномалия;
  на golden редок. Не «чиним», а показываем.
- Частичный/съехавший `outline` в чужих файлах не верифицирован за пределами golden — `file_has_outline`
  и fallback на глубину-кода это закрывают (без outline теряем ≤27/809, см. §1).

## 9. Критерии приёмки

**Этот патч (§2–§4 + домен):**
- `resolve_ancestor_indices(depths)`, `node_depth`, `leaf_flags`, `canonical_codes`, детектор
  аномалий реализованы как чистые доменные функции; инвариант монотонности проверяется property-тестом.
  (`canonical_codes` без экспортного потребителя — он в отложенном §5.)
- Парсер читает `outline_level` (openpyxl), выровнен по `source_index`; `file_has_outline` корректен;
  fallback на глубину-кода работает на плоском файле.
- **🔴 Ассерт выравнивания** openpyxl[`si+2`] == pandas[`si`] стоит на каждом узле и падает громко;
  «грязный» тест подтверждает, что он ловит рассинхрон.
- Три точки резолва используют один механизм; `is_leaf` — позиционный; сервис-тест подтверждает
  разный `embedding_input` на коллизии.
- `ParsedEstimate.anomalies` (построчные) + `outline_overrides` (агрегат) текут в ответ
  `POST /estimates`; фронт показывает построчную таблицу + агрегатную строку про outline; по
  аномалиям — «по коду X → взяли Y».
- Пост-замер `eval_matching` снят (top-1/top-3 + срез `10.2.1.x`/газон + счётчики ≈68/≈115),
  результат записан → **дерево подтверждено** (предусловие для §5).
- `just test` зелёный; `uv run ruff check .` чист.

**Отложенный патч (§5, после подтверждения дерева):**
- Экспорт: столбец «Замечания по структуре» всегда; галочка «Исправить нумерацию» перезаписывает
  `№ раздела` каноническими кодами и выносит оригинал в замечания; без галочки `№ раздела` не тронут;
  экспортный round-trip-тест зелёный.
