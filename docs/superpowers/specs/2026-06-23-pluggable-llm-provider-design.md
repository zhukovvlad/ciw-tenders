# Спецификация: переключаемый LLM-провайдер арбитра матчинга

**Дата:** 2026-06-23
**Статус:** дизайн утверждён, готов к плану реализации

## Место в общей картине

Отдельное улучшение поверх SP2 (асинхронный матчинг). **Не часть SP3** (ревью/правки + запись
`Статья СМР` + выгрузка) — SP3 не трогаем. Меняется только адаптерный слой LLM-арбитра; порт
`LLMMatcher` и логика `MatchingService.match_one` (порог → арбитр → честный score) остаются как есть.

## Задача

Сейчас LLM-арбитр (выбор статьи из топ-K при `score ≤ 0.90`) жёстко завязан на прямой Anthropic
SDK (`AnthropicLLMMatcher`, ключ `ANTHROPIC_API_KEY`). Нужно сделать провайдера **переключаемым по
конфигу**: OpenRouter (новый, дефолт) или Anthropic (текущий, опция). Архитектура должна позволять
добавить Google как +1 адаптер без изменения порта/сервисов. Эмбеддинг (`OpenRouterEmbedder`) —
**вне объёма**, остаётся как есть.

## Вне объёма

- Эмбеддер (узлов/справочника) — остаётся OpenRouter, не делаем переключаемым.
- Google (Gemini) адаптер — не сейчас; абстракция оставляет его добавлением одного файла.
- Любые правки SP3.

## Архитектура (подход A)

Порт `LLMMatcher.choose_best(query, candidates) -> TemplateArticle | None` — **без изменений**.

**Общий модуль** `backend/app/infrastructure/ai/llm_matching_common.py` — единая семантика арбитража,
независимая от провайдера (от неё зависят «честный score» и трактовка отказа, поэтому она ОДНА):
- `SYSTEM_PROMPT` — константа. Жёстко требует: «ответь ТОЛЬКО номером строки `1..K` (НЕ кодом статьи),
  без слов; если ни один кандидат не подходит — ответь `0`». Канал отказа (`0`) **обязателен** в промпте,
  иначе у модели нет способа сказать no-match и она вынужденно выберет плохого кандидата (а `parse_choice`
  никогда не отработает по назначению).
- `build_user_prompt(query, candidates) -> str` — нумерованный список **только имён** (`1. name`),
  **без `[code]`**. Код статьи арбитру для выбора не нужен (выбор по семантике имени), а его показ
  провоцирует модель вернуть код вместо номера строки → misroute (см. «Крайние случаи»). Код
  восстанавливается по индексу уже после выбора.
- `parse_choice(text, candidates) -> TemplateArticle | None` — извлекает **первый целочисленный токен**
  (`re.search(r"\d+")`, не равенство — устойчиво к «option 2»/markdown); `0`/нет числа/вне диапазона
  `1..K` → `None` (отказ). Возвращает `candidates[choice-1].article`.
  **Наблюдаемость:** различать легитимный отказ и сбой формата — если ответ непустой, не `"0"`, и не
  даёт валидного индекса, писать **warning в лог** (для тюнинга промпта/модели). Возврат остаётся
  `None` в обоих случаях (порт не меняется), но в логах видно «модель вернула нечитаемое», а не глухой
  no-match.

**Адаптеры** (оба тонкие: только вызов провайдера + классификация ошибок; промпт/парсинг — из common):
- `AnthropicLLMMatcher` (рефактор) — прямой `anthropic` SDK; переводим на `llm_matching_common`,
  добавляем `temperature=0` (сейчас не задан → дефолт 1.0, недетерминированно).
- `OpenRouterLLMMatcher` (новый) — OpenAI-совместимый `POST {openrouter_base_url}/chat/completions`
  через `httpx` (зеркалит `OpenRouterEmbedder`: инъекция `client` для тестов, `retry_transient` +
  `_is_transient`). `temperature=0`. Заголовки атрибуции OpenRouter `HTTP-Referer` и `X-Title`.
  Переиспользует `OPENROUTER_API_KEY` (новый ключ не нужен).

**Фабрика** `get_llm_matcher()` в `deps.py` — **только строит** адаптер по `settings.llm_provider`.
Валидация (провайдер/ключ/депрекация) живёт в конфиге (`@model_validator`, ниже), а НЕ в фабрике:
фабрика — FastAPI-зависимость (`Depends`), исполняется на первом запросе, а не при подъёме приложения,
поэтому проверки в ней — не fail-fast. Валидатор конфига срабатывает при инстанцировании `Settings`
(т.е. при импорте через `get_settings()` — общий путь и для FastAPI, и для Celery-воркера) → настоящий
fail-fast на старте процесса. Неизвестный `llm_provider` в фабрике → `ValueError` как страховка (но до
неё дело не дойдёт — конфиг уже упадёт).

## Конфиг (`backend/app/core/config.py`)

- `llm_provider: str = "openrouter"` — `"openrouter"` | `"anthropic"`.
- **Per-provider модели** (переключение = флип одной `LLM_PROVIDER`; слаги в разных неймспейсах):
  - `openrouter_llm_model: str = "anthropic/claude-3.5-sonnet"` — слаг OpenRouter (⚠ сверить актуальный
    перед мерджем, слаги меняются).
  - `anthropic_llm_model: str = "claude-3-5-sonnet-20240620"` — нативный Anthropic id.
- `openrouter_base_url: str = "https://openrouter.ai/api/v1"` — только для OpenRouter-матчера
  (имя по провайдеру, как существующий `embedding_base_url`; SDK-адаптер Anthropic его игнорирует).
- `GOOGLE_API_KEY` — оставляем зарезервированным под будущий Google-адаптер (не «мёртвый», а «впрок»).

**Единый `@model_validator(mode="after")` в `Settings` — настоящий fail-fast (срабатывает при
инстанцировании конфига, т.е. на старте и FastAPI, и Celery-воркера):**
1. **Депрекация `llm_model`:** поле удаляем; `extra="ignore"` молча проглотил бы старую `LLM_MODEL`,
   поэтому валидатор проверяет `os.environ` напрямую и **роняет старт**: «`LLM_MODEL` устарел → задайте
   `OPENROUTER_LLM_MODEL` и/или `ANTHROPIC_LLM_MODEL`». (Fallback-чтение отвергнуто: старый Anthropic-слаг
   в OpenRouter — некорректен → это и есть тот misroute, который фича исключает.)
2. **Неизвестный `llm_provider`** (не `openrouter`/`anthropic`) → `ValueError` с понятным сообщением.
3. **Отсутствует ключ выбранного провайдера** → `ValueError` на старте: `openrouter` требует
   `OPENROUTER_API_KEY`, `anthropic` — `ANTHROPIC_API_KEY`. Проверка считает **пустую строку
   отсутствием** (pydantic-дефолт ключей — `""`, не `None`): условие — `if not value`.

> Тесты: `conftest` уже кладёт `OPENROUTER_API_KEY`/`ANTHROPIC_API_KEY` в `os.environ`, поэтому валидатор
> проходит в общем прогоне. Тест на «missing-key → `ValueError`» использует `monkeypatch.delenv` для
> нужного ключа; тест на депрекацию — `monkeypatch.setenv("LLM_MODEL", ...)`.

## Поведение и крайние случаи

- **Модель вернула КОД статьи вместо номера строки (реальный misroute):** домен — «Статья СМР» с
  числовыми/точечными кодами (`08.03.01`). Если показать код в списке, модель может вернуть его, а не
  индекс; `re.search(r"\d+")` на «08.03.01» даст `8` → вне диапазона (`top_k`=3) → ложный `no_match`,
  или (при большом K) тихо не тот кандидат. Парсингом не лечится (валидный индекс от кода не отличить).
  Лечение: (1) **не показывать `[code]`** в `build_user_prompt` (см. common); (2) промпт явно требует
  «НОМЕР СТРОКИ `1..K`, не код». Тест-кейс «модель вернула код» обязателен.
- **Разговорчивость моделей:** через OpenRouter роутимся на разные модели с разной дисциплиной вывода.
  Защита от болтливости — `temperature=0` + строгий промпт «только число» (+ `parse_choice` берёт первый
  int, устойчив к «option 2»/markdown). `max_tokens` болтливость НЕ снижает — он обрезает вывод; для
  несговорчивой модели маленький лимит может срезать саму цифру → ложный `None`. Остаточный риск
  («между 1 и 3 выбираю 2» → первый int = 1) приемлем при temp=0 + строгом промпте; не усложняем.
- **Ограничение по модели (записать явно):** `max_tokens` мал (≈16, дешёвый кэп) — это предполагает
  **обычную chat-модель**. **Reasoning-модель не поддерживается без доработки:** её внутреннее
  рассуждение съест бюджет токенов → пустой `content` → ветка «нет choices» → ошибка → `partial_error`
  на каждом узле. Pluggability подразумевает chat-слаги; reasoning требует иной трактовки `max_tokens`
  (вне объёма). Зафиксировать в README/.env-комментарии, чтобы пользователь не словил загадочный
  `partial_error`.
- **Отказ:** `0` / не-число / вне диапазона → `None` → `match_one` трактует как `no_match` со снимком.
- **Галлюцинация** (модель назвала статью вне топ-K): индекс вне `1..K` → `None`; плюс существующая
  валидация в `match_one` (выбранный обязан быть из кандидатов) остаётся второй линией.

## Транзиентность и ошибки OpenRouter в теле ответа

Два независимых `_is_transient` (Anthropic — типизированные исключения SDK; OpenRouter — httpx +
status code) допустимы, но должны покрывать **одни и те же логические условия**: сеть/таймаут, 429, 5xx.

**OpenRouter 200 с ошибкой в теле** (не ловится по status code) — адаптер инспектирует JSON и
**разделяет по типу ошибки, а не трактует всё как транзиент**:
- **Транзиентное** (rate-limit в теле, upstream 5xx) → `TransientError` (ретрай по бюджету; при
  исчерпании — узел `error` → `partial_error`).
- **Перманентное** (невалидный слаг модели / model-not-found, auth, context-length, модерация) →
  **громкий не-транзиентный exception** (НЕ `TransientError`): крутить весь бюджет ретраев бессмысленно.
  Он пробросится через `match_one` → `_match_nodes` (ловит только `TransientError`) → top-level `except`
  в `match_estimate` → `partial_error` + видимый фейл задачи в логах Celery. Невалидный слаг — **самая
  вероятная** реальная поломка (см. ⚠ «сверить слаг»), поэтому её нельзя глушить в retry→partial.
- **В любом случае логировать тело `error`** (operator видит «model not found», а не глухой
  `partial_error`).
- Возврат `None` тут НЕ используется (иначе ложный `no_match`).

## Тестирование (фейки, без сети)

- `parse_choice` (common): `"2"`→кандидат 2; мусор/вне диапазона→`None`; **обёрнутый ответ**
  («The best match is option 2.» / markdown) → кандидат 2; **ответ-код** («08.03.01» при `top_k`=3) →
  `None` (вне диапазона, не «не тот кандидат»); warning в логах на непустой-непарсящийся ответ.
  - **`"0"` → `None` (отказ), НЕ `candidates[-1]`** — явный тест на off-by-one: `0` должен отсекаться
    условием (`choice == 0 or not 1 <= choice <= K`) ДО индексации `candidates[choice-1]`, иначе
    `candidates[-1]` молча вернёт последнего кандидата = ложный матч вместо отказа.
  - **`"0"` НЕ пишет warning** — легитимный отказ ≠ сбой формата; warning только на непустой-не-`"0"`-
    непарсящийся ответ.
- `build_user_prompt`: листинг **без кода** (только имена) — проверить, что код в промпт не утекает.
- `OpenRouterLLMMatcher` с инъецированным фейковым `httpx`-клиентом: успешный `"2"`→кандидат;
  transport/timeout→ретраи→`TransientError`; **200 с транзиентной ошибкой в теле→`TransientError`**;
  **200 с перманентной ошибкой (model-not-found/auth)→громкий не-транзиентный exception** (не `None`,
  не `TransientError`); отправляются заголовки атрибуции.
- `AnthropicLLMMatcher` (после рефактора) — поведение не изменилось (тот же парсинг через common; добавлен
  `temperature=0`).
- Конфиг-валидатор: неизвестный `llm_provider`→`ValueError`; **отсутствует/пустой ключ выбранного
  провайдера→`ValueError`** (через `monkeypatch.delenv`); **задана `LLM_MODEL`→`ValueError`** (депрекация).
- Фабрика `get_llm_matcher()`: правильный тип адаптера на каждый `llm_provider`.

## Журнал решений (кратко)

1. Подход **A** (адаптеры за портом + общий `llm_matching_common`) — гарантирует идентичную семантику.
2. Провайдеры сейчас: **OpenRouter + Anthropic**; Google — позже одним адаптером.
3. Дефолт — **OpenRouter** (переиспользует `OPENROUTER_API_KEY` эмбеддера).
4. Модели — **per-provider** переменные (слаги в разных неймспейсах; один общий `LLM_MODEL` ломает
   переключение).
5. Старый `LLM_MODEL` — **fail-fast на старте** (не тихий игнор, не fallback-misroute).
6. Fail-fast — в **`@model_validator` конфига** (не в фабрике-`Depends`, та бы сработала лишь на первом
   запросе): неизвестный провайдер + отсутствующий/пустой ключ + депрекация `LLM_MODEL`.
7. `temperature=0` на обоих адаптерах; `parse_choice` — первый int.
8. **Код статьи не показываем** в листинге кандидатов (промпт требует номер строки, не код) — закрывает
   misroute «модель вернула код».
9. OpenRouter 200-с-error-телом: **транзиент→`TransientError`, перманент→громкий exception** (не глушим
   невалидный слаг в retry→partial); всегда логируем тело ошибки.
10. `max_tokens` мал — **только обычные chat-модели**; reasoning вне объёма (документируем).
