# Дизайн: фильтрация организационных заголовков в потоке смет (lean v1)

**Дата:** 2026-06-25
**Статус:** черновик на ревью
**Связано:** [2026-06-22-estimate-hierarchical-parser-design.md](2026-06-22-estimate-hierarchical-parser-design.md)

## Проблема

В дерево узлов сметы попадают строки двух разных природ:

- **Виды работ** — реальные работы или их разделы («Земляные работы», «Устройство
  кровли»). Их нужно сопоставлять со справочником.
- **Организационные заголовки** — узлы, чьё наименование описывает не работу, а
  структурную единицу проекта: этап, очередь, пусковой комплекс, корпус, литер,
  секция, объект. Пример из реальной сметы — строка «I и 2 Этапы БЦ и ЖК».

Организационные заголовки семантически «пустые» в терминах работ. При эмбеддинге
они сваливаются в общий строительный центроид и притягиваются к всеядной статье
справочника «Прочее (подготовительные работы, содержание площадки)» с обманчиво
высокой близостью (~0.86), засоряя таблицу результатов и заставляя оператора
вручную отбраковывать мусор.

Дополнительно: крошка `embedding_input` узла собирается из имён предков
([estimate_parser.py:76-84](../../../backend/app/services/estimate_parser.py#L76-L84)).
Оргось стоит почти под каждым корнем дерева, поэтому оргтокены предков попадают в
вектор **выживших работных** узлов — одностороннее загрязнение (справочник
оргтокенов не содержит, см. «Проверки на данных»), тянущее косинус правильных пар
вниз.

**Ключевое:** организационность — свойство СМЫСЛА имени, а НЕ позиции в дереве.
Узел «Земляные работы» тоже имеет детей, но это работа. Структурный признак «есть
дети» не годится.

## Решение (обзор)

До эмбеддинга классифицировать каждый узел по смыслу имени в класс
`WorkClass ∈ {WORK, ORG, UNSURE}` каскадом «дёшево → дорого». Класс —
**вычисляемый в памяти**, отдельной колонки в БД нет (см. «Модель данных»). Он
питает **двух потребителей**:

1. **Гейт эмбеддинга/матчинга:** чистый орг-заголовок (`ORG`) обратимо исключается
   — ему проставляется терминальный `status='excluded'` (не эмбедится, не матчится,
   строка не удаляется, в столбец «Статья СМР» при выгрузке — пусто).
2. **Сборку крошки:** при построении `embedding_input` для выживших узлов
   ORG-предки выбрасываются.

Орг-заголовок **не получает класса сопоставления** (никакого `confident`/
`needs_review`/`no_match`) — он структурный, а не результат матчинга. Дефолт при
любом сомнении — «оставить» (`UNSURE` идёт в эмбеддинг/матчинг как WORK). Асимметрия
ошибок: ложный ORG (молчаливый пропуск работы) дороже ложного WORK (оператор
отбракует на проверке).

### Модель данных: только `status='excluded'`, без `node_class`

Класс `WorkClass` нужен лишь в проходе классификации (для выреза ORG-предков из
крошки) и **не персистится**. Единственный след в БД — терминальный
`status='excluded'`. Это «никакого класса»: орг-строка несёт признак «исключена», а
не отдельную ось класса. Колонка `status` уже `VARCHAR` без CHECK-констрейнта →
новое значение `'excluded'` **не требует миграции**. `WORK`/`UNSURE` поведенчески не
различаются (оба матчатся) — отдельный маркер им не нужен.

**`excluded` ставится для ORG из ЛЮБОГО источника** — и лексической ветки row-2
(чистый каркас), и **вердикта LLM** на ветке смеси. Иначе рассинхрон: класс узла
используется для выреза предков из крошки потомков, поэтому узел, признанный ORG,
обязан и исключаться из матчинга, и вырезаться из крошки — единым решением. Финальный
класс (после LLM) — единственный источник истины для обоих потребителей.

Следствия, принятые осознанно:
- **`excluded` недетерминированен между прогонами.** Лексические исключения (в нашем
  корпусе 48) стабильны, но LLM на ~103 смешанных именах при повторном запуске задачи
  может флипнуть узел в/из `excluded` (классификация гоняется заново каждым прогоном).
  Для stateless-потока это приемлемо; персистентного кэша, который бы это стабилизи­
  ровал, в v1 нет (phase 2).
- **Нет следа `WORK` vs `UNSURE` в БД.** Постфактум-аналитики «что трогал LLM» не
  будет до phase 2 (там всё равно перегон). Осознанная потеря, не бесплатный обед.
- **Фронт трактует `status='excluded'` ⇒ структурная орг-строка** — корректно, **пока
  у `excluded` единственный производитель — эта фича**. Появится второй источник
  `excluded` — рендер «как орг» придётся пересмотреть.
- **Обратимость** = переход машины состояний `excluded → pending` + повторный
  embed/match. Данные это поддерживают (строка жива), но сам переход — путь phase 2.

## Каскад классификации

| Условие имени | Класс | LLM? |
|---|---|---|
| Оргтокена в имени нет | `WORK` | нет |
| Есть оргтокен, **нет** работной головы | `ORG` | нет |
| Есть оргтокен **и** работная голова (смесь) | `UNSURE` → LLM | да |

### Оргсловарь (маленькое закрытое множество)

`этап, очередь, пусковой комплекс, корпус, литер(а), секция, блок-секция, объект`
+ организационные аббревиатуры `ЖК` (жилой комплекс), `БЦ` (бизнес-центр).

**Морфология обязательна.** Оргтокены матчатся по стему с русскими окончаниями:
`этап → этапы / этапов / этапе`, `корпус → корпусов / корпуса`,
`очередь → очереди`. Иначе founding-пример «Этап**ы**» и «корпус**ов** 5» утекают
(проверено: наивная граница слова их пропускает). Стемы **заякорены на начало
слова** (`\bэтап[а-яё]*`, не вхождение в середину). Короткие стемы (`секци`,
`литер`, `объект`) могут цеплять работные слова («секционные перегородки») — это
безопасно (попадает в смесь→LLM, не в молчаливый ORG), лишь добавляет объём LLM;
якорь на начало слова это удерживает.

**Аббревиатуры `ЖК`/`БЦ` матчатся литералами, не стемом** (у них нет окончаний).
Это критично из-за пересечения с правилом «аббревиатура = голова» (ниже): литералы
`ЖК`/`БЦ` обязаны распознаваться как оргтокены **раньше** аббревиатурного правила,
иначе чистый каркас «1 Этап ЖК» протечёт в LLM (течь стоимости, не корректности).

**Жёсткое ограничение:** `этап ≠ этаж`. «этаж / этаже / этажность» — это работный
квалификатор уровня, НЕ оргтокен. Тест-кейсы обязательны.

**НЕ добавлять `здание`** — ловит ложные срабатывания на «…подземной части
здания» (это работа). `объект` оставляем, но он безопасен только из-за границы
слова + проверки работной головы («Объект озеленения» → есть голова → смесь→LLM,
не молчаливый ORG); закрыть тест-кейсом.

### Работная голова — определяется НЕГАТИВНО

Не строить белый список работных слов (их открытое бесконечное множество).
`has_work_word(name)` через отрицание:

- токенизировать имя по буквам;
- **сначала** отсечь оргтокены — по морфологическому стему **и** по литералам
  `ЖК`/`БЦ` — и стоп-слова; этот шаг идёт **до** аббревиатурного правила, иначе
  `ЖК`/`БЦ` (тоже 2 заглавные) ложно сочтутся головой;
- оставшийся токен — работная голова, если (длина ≥ 3 букв **ИЛИ** это
  аббревиатура — 2–4 заглавных буквы, рус/лат);
- любая работная голова ⇒ `True`.

Единственный вручную поддерживаемый словарь — оргсловарь + короткий стоп-лист
(предлоги, «прочее», «и т.п.», чистые числа/коды/единицы). Никакого корпуса работ.

**`has_work_word` обязан ошибаться в сторону «голова есть».** Поэтому
аббревиатуры (2–4 заглавные: МАФ, СОФ, ЗИП, КС, ПУТ, VRF) считаются головой — это
закрывает единственную точку молчаливого пропуска работы: «МАФ Корпус 3»,
«ЗИП 1 Этап» → есть голова → смесь→LLM, а не ORG-без-LLM.

> **Отвергнуто (данные).** Дополнительный «length-софтенинг» (короткое имя →
> UNSURE) не нужен и вреден: он загонял в LLM чистый каркас «Этап 1»/«Корпус 1».
> Усиленного `has_work_word` достаточно. Берём только усиление.

### Шорткат `Вид раздела = СМР` — НЕ применять

В нашем парсере `section_type` снимается с узла глубины 1 и **наследуется всем
потомкам** ([estimate_parser.py:71-74](../../../backend/app/services/estimate_parser.py#L71-L74)),
поэтому «Вид=СМР» несёт каждый узел под СМР-корнем и шорткатом WORK быть не может.
Он и не нужен: СМР-корни («Земляные работы») оргтокенов не несут → и так `WORK` по
строке 1 каскада.

## Размещение в пайплайне

Классификация — шаг в async-задаче матчинга
([estimate_matching_service.py](../../../backend/app/services/estimate_matching_service.py)),
ДО эмбеддинга (LLM нельзя тащить в чистый парсер). Поток:

```
lock → RUNNING → classify(все узлы) → rebuild embedding_input (выброс ORG-предков)
     → embed (WORK/UNSURE) → gate → match (WORK/UNSURE) → status
```

## Слои (Clean Architecture)

- **domain:**
  - `WorkClass(str, Enum)` = WORK/ORG/UNSURE в
    [entities.py](../../../backend/app/domain/entities.py).
  - Константы: оргсловарь, стоп-лист, морфологический регэксп оргтокена.
  - `classify_lexical(name) -> WorkClass` и `has_work_word(name) -> bool` — чистые
    функции.
  - `build_embedding_input(name, ancestors_with_class) -> str` — крошка с выбросом
    ORG-предков, `collapse_repeats`, нормализацией пробелов.
  - Порт `WorkTypeClassifier` в [ports.py](../../../backend/app/domain/ports.py).
- **infrastructure/ai:** адаптер LLM-классификатора рядом с арбитром —
  **дешёвая модель (Haiku-класс, конфигурируемо, отдельно от арбитра матчинга)**,
  батч (30–40 имён), few-shot из реальных строк, строгий JSON, async с лимитом
  конкуренции. Вход элемента батча: имя + цепочка предков (контекст).
- **services:** оркестрация каскада (лексика → отбор смеси → LLM), простановка
  `status='excluded'` чистым орг-заголовкам, пересборка крошки выживших.
- **db:** **без новой колонки**. ORG-строки получают `status='excluded'`. Правится
  **только** `fetch_unembedded_nodes` — он отбирает по `embedding IS NULL`, поэтому
  без явного `status != 'excluded'` подхватил бы орг-строки (эмбеддинга у них нет) →
  нужен фильтр. `fetch_matchable_nodes` **проверен в коде**
  ([estimate_repository.py:230-241](../../../backend/app/infrastructure/db/estimate_repository.py#L230-L241)):
  отбирает по `status.in_(('pending','error','no_match'))` И `embedding IS NOT NULL`
  → `excluded` отсекается дважды, правка не нужна. Других запросов «… IS NULL»-стиля,
  выбирающих работу, нет. `nodes_count` считает `status != 'excluded'`.
- **api:** новое поле не нужно — фронт отличает орг-заголовки по `status='excluded'`
  (показывает их структурно, без вердикта; «Статья СМР» пуста).
- **export:** `EstimateExportService._cell_value` уже отдаёт `""` для всего, кроме
  `confident`/`confirmed`/`overridden` → `excluded` экспортится пустой ячейкой без
  правок (зафиксировать характеризующим тестом).

## Чистка крошки (в v1, не выделять)

Это половина ценности фичи, причём более ценная: исключение ORG чистит таблицу
результатов, а чистка предков чинит косинус **основной массы выживших работ**
(оргось — почти под каждым корнем). Почти бесплатна (метка уже есть, классификация
и так до эмбеддинга) и безрисковна (детерминированная функция от готовых меток;
узел жив, меняется лишь его крошка).

- Получают чистку и `WORK`, и `UNSURE` (последние эмбедятся). Классы предков
  известны из прохода классификации (в памяти) — персистить их для этого не нужно.
- Сохранить `collapse_repeats` (повторы сегментов) + нормализацию пробелов
  (двойные пробелы во входных данных) — они валидированы на реальных данных и
  влияют на косинус.
- Позиционный срез оргквалификатора из **собственного** имени узла — phase 2 (это
  рискованная добавка, граница «ядро / добавка» проходит здесь).

## Обработка ошибок

LLM-сбой / битый JSON → весь затронутый батч → `UNSURE` (keep), **никогда ORG**.
Узлы идут в эмбеддинг/матчинг как обычно, ничего не теряется, статусы не трогаются
(как уже устроено для транзиентов в `_embed_nodes`/`_match_nodes`).

## Интеграционный стык: счётчики готовности и `nodes_count`

ORG-узлы исключены из эмбеддинга и матчинга → они никогда не получат матч-статус.
Это ломает два счётчика, если их не поправить:

- **Знаменатель завершения.** `count_unfinished_nodes` сейчас считает строки со
  `status='pending'`
  ([estimate_repository.py:306-314](../../../backend/app/infrastructure/db/estimate_repository.py#L306-L314)).
  ORG-узлы остались бы `pending` навсегда → смета зависнет в `partial_error`.
  Решение: при классификации проставить ORG `status='excluded'`. Тогда
  `count_unfinished_nodes` (`status='pending'`) и `count_node_errors`
  (`status='error'`) их **автоматически** не считают — отдельная правка фильтра не
  нужна. Завершение (`ready`/`partial_error`) считается **только по WORK/UNSURE**.
- **`nodes_count` во фронтовом списке.** Соседняя спека списка смет отдаёт
  `nodes_count` = число **всех** строк
  ([estimate_repository.py:106-130](../../../backend/app/infrastructure/db/estimate_repository.py#L106-L130)).
  После этой фичи число двусмысленно (все vs матчируемые). Зафиксировать: считаем
  `status != 'excluded'` — показываем **матчируемые** (WORK/UNSURE). Решить сейчас,
  пока спеки рядом.

## Зазор имён-объектов — принят осознанно

Имя-объект без оргтокена («Гостиница Заря») лексика оставит как `WORK`. Это
безопасно не только по асимметрии: такой узел всё равно идёт в матчинг, ловит
низкую близость (магнит «Прочее» ~0.86 < 0.90) → попадает к арбитру → метка
«требует проверки», где оператор его и видит. **Он не теряется молча** — всплывает
в штатной корзине на проверку. Поэтому отложенный «монитор утечек» (phase 2) v1 не
блокирует. На данных этот зазор почти пуст (см. ниже).

## Тесты (pytest)

- **Чистый domain:**
  - `has_work_word`: аббревиатуры (МАФ, ЗИП, VRF) → True; «прочее», предлоги,
    числа → не голова.
  - Оргтокен-регэксп: «Этапы», «корпусов 5», «очереди» → True; «этаже»,
    «этажность», «…части здания» → False.
  - `classify_lexical`: «Корпус № 2; 3; 4» → ORG (без LLM); **«1 Этап ЖК» → ORG
    (без LLM)** — точка пересечения «ЖК-литерал» и «аббревиатура=голова», зафик­
    сировать тестом; «Наружное освещение 1 Этап ЖК» → UNSURE(смесь);
    «Устройство кровли» → WORK; «МАФ Корпус 3» → UNSURE (не ORG);
    «Объект озеленения» → UNSURE (есть голова, не молчаливый ORG).
  - `build_embedding_input`: ORG-предки выброшены; `collapse_repeats` и
    нормализация пробелов работают.
- **LLM-адаптер:** мок клиента — батчинг, парсинг строгого JSON, фолбэк в UNSURE
  на ошибке/битом JSON.
- **services:** фейки портов — чистый каркас получил `status='excluded'` и
  исключён из embed/match/счётчиков, крошка выживших без оргтокенов предков.
- **export:** характеризующий тест `_cell_value` для `status='excluded'` → `""`.

## Вне области (YAGNI, phase 2+)

- Персистентный `ClassificationCache` + сид из справочника + версионирование.
  Не нужен в v1: лексика маршрутизирует ~87% детерминированно ((658+48)/809), в LLM
  уходит ~13% — ~100 имён на смету (~3 батча), стоимость незначительна.
- Цикл обратной связи оператора → few-shot; монитор утечек в «Прочее».
- Позиционный срез оргквалификатора из собственного имени узла.
- Фильтр не-СМР разделов (оборудование и т.п.) — отдельная тема.
- UI-действие «вернуть из корзины» (данные это поддерживают — строку не удаляем;
  кнопка — отдельный мелкий слайс).

## Definition of Done

- Каскад классифицирует все узлы (в памяти); чистый каркас («1 Этап ЖК»,
  «Корпус № 2; 3; 4») → `ORG` лексикой без LLM → `status='excluded'`; смесь
  («…1 Этап ЖК») → `UNSURE`
  или WORK, **никогда** ошибочный ORG.
- `embedding_input` выживших узлов **не содержит** оргтокенов из предков.
- ORG-узлы получили `status='excluded'`, исключены из эмбеддинга/матчинга/счётчиков
  **обратимо** (не удалены); «Статья СМР» при выгрузке пуста.
- LLM-фолбэк: битый JSON → UNSURE, не ORG.
- **Гейт данных №1:** оргтокены по справочнику = 0 ложных (справочник org-free).
  ✅ Проверено: 0 из 362.
- **Гейт данных №2 (область — только лексическая ветка):** ветка «ORG без LLM» не
  содержит аббревиатур-работ. ✅ Проверено на выгрузке: 48 узлов, все — чистый каркас
