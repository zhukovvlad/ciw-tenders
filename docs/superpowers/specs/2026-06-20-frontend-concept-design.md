# Дизайн: Концепт фронтенда (CIW · MR Design System)

**Дата:** 2026-06-20
**Статус:** концепт утверждён, ожидает плана реализации
**Связанные файлы:** `frontend/` (Vite + React + TS + Tailwind v4 + shadcn/ui),
`temp/design_handoff_mr_design_system/` (источник токенов и шрифтов)

## Цель

Переосмыслить фронтенд приложения как цельный продукт под фирменную **MR Design
System** (тёмная фиолетово-чёрная тема). Это **концепт**: визуальный язык + набор
ключевых экранов с их состояниями и поведением, «как это должно выглядеть и
ощущаться». Документ фиксирует решения; точная реализация (компоненты, маршруты,
API-контракты) — в последующем плане.

## Кого обслуживаем

Главный герой — **сметчик-оператор**: грузит сметы одну за другой, быстро
пролетает сотни строк, фокусируется на спорных и правит их. Ключевые качества
интерфейса: **скорость** и **точность на больших таблицах**. Всё остальное
(эстетика, анимации) — подчинено этому.

## Контекст и ограничения (не нарушать)

- **Stateless на сервере.** Сметы нигде не хранятся на бэкенде: загрузил →
  распарсили (отбор строк «Вид раздела = СМР») → обогатили из справочника → вернули
  результат. Персистентен только справочник (`template_articles`).
- **«Нет истории» ≠ «сессия одноразовая».** Это два разных понятия, и их нельзя
  слеплять (см. раздел «Долговечность сессии»). Долгосрочной истории обработанных
  смет нет — но **текущая работа оператора не должна теряться** при перезагрузке/
  случайном уходе со страницы. Результат прогона (эмбеддинги + LLM-арбитраж по
  сотням строк) **стоит денег и времени**, и потеря его на середине ревью — это не
  косметика, а «оператор потерял час и оплаченный прогон».
- **Логика сопоставления** (бэкенд, не меняем): эмбеддинг → топ-3 (pgvector) →
  `score > 0.90` ⇒ «Уверенное совпадение»; иначе LLM-арбитр (Claude) выбирает из
  топ-3 ⇒ «Требует проверки»; ничего подходящего ⇒ «Нет совпадения».
- **API уже отдаёт топ-3 кандидата** со score по каждой строке
  (`MatchResult.candidates`). Значит раскрытие строки и альтернативы в Excel
  **не требуют отдельного прогона** — данные приходят сразу.
- **Авторизация** (бэкенд готов): роли `user`/`admin`. В концепте рисуем **только
  экран входа**; ролевых различий в UI пока **не вводим** — все залогиненные видят
  одно и то же.
- **Выгрузка результата** на бэкенде сейчас возвращает JSON, не `.xlsx`. Это
  **критический путь** (без него цикл не замыкается — проверять есть что, а деть
  результат некуда), а не «задача потом». Подробности — в разделе «Формат выгрузки»
  и «Зависимости от бэкенда» (эндпоинт выгрузки — первый в очереди реализации).

## Визуальный язык (MR Design System)

Источник правды по значениям — `temp/design_handoff_mr_design_system/tokens/`.
В реализации токены MR DS вытесняют текущую стоковую светлую тему shadcn.

| Аспект | Решение |
|---|---|
| Тема | Тёмная. Фон `--ds-ink #0A0814`, поверхности `surface/surface-2`, текст `--ds-text #F4F2FA`. |
| Акцент | Violet `#754AE8` (действия), hover `#8B66F0`, active `#5733C0`. |
| Состояния | success — mint `#82D6CC`, warning — tan `#BD9375`, danger — `#E8657A`. |
| Шрифты | Display — **Grtsk Giga** (заголовки, крупные числа); текст/интерфейс — **Suisse Intl**. Самохостинг из `fonts/`. |
| Геометрия | Острая и сдержанная: контролы radius 4px, карточки 8–10px. Не «округлять» сильнее. |
| Движение | Минимум: переходы цвета 0.15s на hover, focus-ring `--ds-focus-ring`. Без декоративных анимаций. |
| Числовые данные | Моноширинный шрифт для кодов статей и score; числовые колонки — выравнивание вправо. |
| Различимость статусов | Три статуса (mint/tan/danger) на тёмном фоне обязаны различаться **беглым взглядом** при быстром скане таблицы. Цвет дублируется иконкой+текстом (для дальтоников ок), но проверить именно скорость различения tan-warning `#BD9375` на ink `#0A0814` от mint/danger; при необходимости усилить форму/позицию метки, не только оттенок. |

Семантические алиасы MR DS (`--ds-bg-*`, `--ds-fg-*`, `--ds-accent*`,
`--ds-state-*`) используются в компонентах вместо примитивов.

## Набор экранов

Единая шапка (shell) на всех экранах после входа: лого `MR · Сметы`, вкладки
**Смета / Справочник**, идентификатор пользователя справа.

### 00 · Вход
Лого по центру, поля логин/пароль (фокус — violet focus-ring), кнопка «Войти».
Под уже готовый auth-бэкенд (email+пароль, JWT). Ошибка входа — текст danger под
формой.

### 01 · Старт / загрузка
Крупная dropzone: «Перетащите смету (.xlsx / .xls)» + выбор файла кнопкой.
- **Состояние drag-over:** рамка и подложка подсвечиваются violet.
- Подпись поясняет: обрабатываются строки «Вид раздела = СМР».
- Ошибка (битый файл / нет строк СМР / не тот формат) — сообщение danger, остаёмся
  на экране старта.

### 02 · Обработка
Пошаговый прогресс одного файла: **отбор строк СМР → векторизация (N/N) → поиск +
LLM-арбитр (k/N)**, с грубой оценкой оставшегося времени. LLM-арбитраж происходит
**здесь** (на этапе обработки), а не во время проверки — на экран проверки
оператор приходит с уже проставленными статусами.

### 03 · Проверка сопоставлений — ГЛАВНЫЙ ЭКРАН
Сердце продукта. Плотная таблица всей сметы.

**Под-панель файла:** имя сметы + счётчики-пилюли (N уверенных / N проверить /
N без пары) + кнопки «＋ Новая смета» и **«↓ Выгрузить Excel»**.

**Тулбар:** фильтр-чипы (**Все / Проверить / Без пары**), индикатор прогресса
проверки **«проверено k из N»** (мотиватор «добить очередь»), поиск по строкам
сметы.

**Таблица** (колонки: `#`, Работа из сметы, Статья справочника СМР, Score, Статус):
- **Уверенные** строки — плотные, с галкой mint и кодом+названием статьи.
- **Требует проверки** — подсвечены полосой tan; в колонке статьи — выбранная
  арбитром статья + «▾».
- **Нет совпадения** — «— без пары —», статус danger.

**Механика правки = раскрытие строки.** Раскрытая спорная строка показывает:
1. **Короткое обоснование арбитра** («почему выбран этот кандидат») — это якорь
   доверия: оператор читает одну строку и за секунду соглашается/нет, а не
   перерешивает задачу арбитра с нуля. Требует доработки бэкенда (см. «Зависимости
   от бэкенда»).
2. **Топ-3 кандидата** (код, наименование), выбранный арбитром выделен
   violet-рамкой. Сырой `score` — **вторичный, приглушённый** (у спорных строк он
   низкий по определению — это близость эмбеддинга, не уверенность выбора;
   крупное низкое число рядом с выбором арбитра дезинформирует).

**Основной рабочий луп — без мыши** (keyboard-first здесь буквальный, не на словах):
оператор приземлился на следующую спорную строку → она **авто-раскрылась** →
видны обоснование и топ-3 → нажал `1 · 2 · 3` (или Enter — подтвердить выбор
арбитра) → строка схлопнулась с галкой, фокус **сам уехал на следующую спорную**.
Отдельная клавиша — «следующая спорная / пропустить». Клик мышью — лишь
альтернативный вход в ту же механику, а не основной путь.

**Escape-hatch:** «＋ Нет верного — искать в справочнике вручную» внутри
раскрытой строки — ручной поиск статьи по справочнику, когда ни один из топ-3 не
подходит (зависит от поискового эндпоинта, см. «Зависимости от бэкенда»).

**«Нет совпадения» и счётчик прогресса.** Нужно различать «оператор посмотрел и
осознанно оставил без пары» (работа не из шаблона — это нормальный исход) и «ещё
не дошёл». Поэтому строка «нет совпадения» тоже **требует подтверждения** оператора
(«ок, без пары»), и счётчик «проверено k из N» считает именно принятые решения
(подтверждённые совпадения + подтверждённые «без пары»). Иначе очередь не
закрывается до нуля и непонятно, баг это или норма.

### 04 · Выгрузка / готово
Крупные числа (Giga): «N сопоставлено» (mint) / «N без пары» (danger). Кнопка
**«↓ Скачать обогащённый .xlsx»** и ссылка **«＋ Загрузить следующую смету»**
(сброс в состояние 01, т.к. долгосрочной истории нет).

**Формат файла специфицируем так же тщательно, как экран 03** — это половина
продукта. Берём исходный Excel и **дописываем колонки справа** (исходные строки и
структура сметы сохранены — оператор узнаёт свой файл):

| Колонка | Содержимое |
|---|---|
| Код статьи | Код выбранной статьи СМР (после правок оператора) |
| Наименование статьи | Название выбранной статьи |
| Score | Косинусная близость (для прозрачности; у спорных низкая) |
| Статус | Уверенное / Требует проверки / Нет совпадения / *подтверждено оператором* |
| Альтернатива 2, Альтернатива 3 | Код+название 2-го и 3-го кандидатов из топ-3 |

Альтернативы рядом дают **дочистку прямо в Excel**: строки «требует проверки»
можно добивать привычными фильтрами/заменой, не возвращаясь в приложение. Данные
для альтернатив уже есть в ответе API (`candidates`), новый прогон не нужен.

### 05 · Справочник статей СМР
Форма добавления (Код / Наименование работы / Раздел + «＋ Добавить») и таблица
статей с удалением. Подпись: при добавлении строка автоматически векторизуется
(Gemini) — это «топливо» поиска. Пустое состояние: «Справочник пуст».

## Долговечность сессии

«Нет долгосрочной истории на сервере» и «сессия восстановима» — **разные вещи**,
их разводим явно. Сервер не хранит сметы; но результат текущего прогона на клиенте
**обязан переживать перезагрузку/случайное закрытие вкладки**, иначе оператор теряет
час работы и оплаченный прогон (эмбеддинги + LLM по сотням строк).

- **Минимум (обязательно):** guard при уходе со страницы (`beforeunload`) и при
  «＋ Новая смета» — подтверждение, если есть непроверенные/несохранённые правки.
- **Лучше (рекомендуется):** держать результат прогона + правки оператора в
  клиентском хранилище, переживающем перезагрузку. Это **не нарушает** «не хранить
  сметы на сервере».
- **Рекомендация по хранилищу:** `sessionStorage` как дефолт — переживает
  перезагрузку и навигацию внутри вкладки, но **сам очищается при закрытии вкладки**
  (никаких потенциально чувствительных смет на диске долгосрочно). Если нужно
  переживать и закрытие вкладки — `IndexedDB`, но тогда явный «выход/очистить» и
  осознанный компромисс по приватности. Финальный выбор — в плане реализации.
- **Ключевой технический вопрос для плана:** что физически держит результат между
  02 → 03 → 04. Сейчас — чистый React `useState` (память вкладки), поэтому риск
  потери реален уже в v1. Слой хранения сессии — часть работы, а не «потом».

## Производительность таблицы

Смета может быть **в тысячи строк** — таблица «всей сметы» **требует
виртуализации** (это требование, не опция). Подводный камень для плана:
виртуализация плохо дружит с **раскрывающимися строками переменной высоты** — многие
готовые библиотеки на этом ломаются. Выбор библиотеки/подхода под «виртуализация +
expandable rows» заложить в план заранее, чтобы не упереться внезапно.

## Принципы поведения

- **Клавиатура — первый класс** на главном экране, буквально: весь луп проверки
  спорных проходится без мыши (авто-раскрытие следующей спорной → `1·2·3`/Enter →
  фокус сам уезжает дальше). Мышь — альтернативный вход, не основной путь.
- **Фокус на спорных:** фильтр «Проверить» + счётчик прогресса ведут оператора по
  очереди спорных строк; очередь закрывается до нуля (каждый исход подтверждается).
- **Доверие к результату через обоснование, не через число:** для спорных строк
  ведём решение коротким «почему» арбитра; сырой низкий score — вторичный.
- **Минимализм действий:** в строке только то, что нужно (выбор/подтверждение). Без
  массовых операций и лишних кнопок в первой версии (YAGNI).

## Зависимости от бэкенда

Концепт фронта опирается на бэкенд; часть уже есть, часть надо доработать.
Порядок — по критичности для замыкания цикла.

1. **🔴 Выгрузка Excel (критический путь, делать первым).** `/estimates/match`
   сейчас отдаёт JSON ([api.ts](frontend/src/lib/api.ts)). Нужен эндпоинт,
   возвращающий обогащённый `.xlsx` по формату из раздела «Формат выгрузки». Без
   него весь цикл не замыкается — фронт упрётся в заглушку. **Первое в плане.**
2. **🟠 Обоснование арбитра.** `LLMMatcher.choose_best(...)` возвращает только
   выбранную статью без «почему» ([ports.py](backend/app/domain/ports.py)). Нужно
   протянуть короткий rationale: порт → Anthropic-реализация → `MatchResult` → DTO
   → API. Без этого экран 03 теряет якорь доверия и замедляет проверку.
3. **🟡 Ручной поиск статьи** для escape-hatch (поиск по справочнику по
   подстроке/коду) — проверить, есть ли эндпоинт, иначе завести.
4. **✅ Топ-3 кандидата — уже есть** (`MatchResult.candidates`). Доработок не
   требует; используется и в раскрытии строки, и в колонках Excel.

## Сознательно отложено

- **Ролевой UI** (`user`/`admin`: кто видит/правит справочник, управление
  пользователями) — auth-бэкенд готов, но в этом концепте только экран входа.
  Добавим отдельным спеком при необходимости.

## Намеренно НЕ делаем (YAGNI)

- Долгосрочную **серверную** историю/хранилище обработанных смет (противоречит
  stateless-архитектуре). Клиентское восстановление **текущей** сессии — НЕ сюда,
  оно как раз делается (см. «Долговечность сессии»).
- Массовые операции над строками, кастомные представления таблицы.
- Ролевые различия в интерфейсе на этом этапе.
- Любые анимации сверх переходов hover/focus из MR DS.
