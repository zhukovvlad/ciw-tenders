# Правка «уверенных» позиций на экране ревью

**Дата:** 2026-07-02
**Статус:** утверждено (брейншторм с оператором)

## Проблема

RAG иногда уверенно (`score > 0.90`) сопоставляет строку с неверной статьёй.
Сейчас `confident`-строки — единственные, которые нельзя раскрыть на экране
ревью (`expandable = flagged || status === "matched_fund"` в `ReviewRow.tsx`),
поэтому оператор не может исправить заведомо неправильную классификацию.
Фонд-хиты (`matched_fund`) уже раскрываются (override через ручной поиск).

## Решение (утверждённый вариант)

Унифицировать: **любая сматченная строка раскрывается кликом** — кандидаты +
ручной поиск + «Оставить без пары». Отвергнутые альтернативы: карандаш на
hover (плохая обнаруживаемость, второй паттерн взаимодействия, мёртв на
тач-устройствах), глобальный «режим аудита» (overkill).

Бэкенд не меняется: `PATCH .../review` уже принимает `pick`/`reject`/`confirm`
для любой строки со статусом ≠ `pending` (`estimate_review_service.py`), а
фонд уже промоутит `overridden`-решения и не рекрутирует обратно подтверждённые
фонд-хиты (`_PROMOTABLE_REVIEW`, анти-накрутка в `decision_fund_service.py`).

## Дизайн (frontend-only)

### 1. Раскрываемость (`ReviewRow.tsx`)

- `expandable` — `true` для всех строк (статус `pending` на экран ревью не
  попадает). Шеврон появляется у уверенных строк, как у остальных.
- Warning-рамка слева остаётся **только** у `flagged = requiresDecision(row)` —
  визуальная иерархия «что требует внимания» не размывается.

### 2. Панель кандидатов

- `confident`: кандидаты уже есть в снимке — панель как у `needs_review`
  (топ-3 + поиск).
- Если текущая рекомендация AI (`matched_code`) отсутствует в `row.candidates`
  (случай фонд-хита, у которого кандидатов нет вовсе), первой строкой панели
  рисуется **синтетическая карточка «текущая рекомендация»**:
  `matched_code` / `matched_name`; вместо score — текстовая метка
  происхождения («Из фонда» с иконкой Database для `matched_fund`,
  «Рекомендация AI» для остальных).
- Клик по синтетической карточке = «вернуть рекомендацию»: `dispatch
  confirmArbiter` + `onReview(row, "confirm")`. Действие `confirm` на бэке не
  требует `article_id` — использует `matched_article_id` строки. Это откат
  случайной правки одним кликом.
- **Инвариант, на котором держится откат:** AI-снимок (`matched_*`,
  `candidates`) иммутабелен — ревью-действия пишут только ось
  `review_status`/`final_*` (`save_review_decision`, спека SP3). Поэтому после
  любого `pick` карточка продолжает показывать *исходную* рекомендацию
  (`matched_code` не меняется и в `candidates` не появляется), а `confirm`
  всегда возвращает именно её. Фонд-хиты имеют `matched_article_id`
  (`estimate_repository.py`, фонд-пасс) — `confirm` для них валиден.

### 3. «Оставить без пары»

Кнопка (`reject`) показывается во **всех** раскрытых панелях, не только у
`no_match`. Кейс: RAG уверенно сматчил строку, которой нет в справочнике.

### 4. Что не меняется

- Прогресс «проверено X из Y» (`requiresDecision`): уверенные/фонд-строки
  по-прежнему не требуют решения.
- Клавиатурная очередь (`queue` в `ReviewScreen.tsx`) — только спорные строки;
  раскрытие уверенной строки мышью отключает хоткеи (как сейчас у фонд-хитов).
- Метки статуса: после правки `statusLabel` сам даёт «Ручной выбор» /
  «Нет совпадения» (логика уже есть, спека фонда §12.4).
- Редьюсер `reviewState.ts` — без изменений.
- Фонд: overridden промоутится, подтверждённый фонд-хит обратно не
  рекрутируется.
- **Reject фонд-хита НЕ гасит запись фонда** (осознанный скоуп): `rejected`
  не входит в `_PROMOTABLE_REVIEW`, фондовая запись остаётся и продолжит
  матчить это имя в следующих сметах. Гашение/понижение фондовой записи по
  отвергнутому хиту — отдельная задача, занесена в `docs/TECH_DEBT.md`.

## Тесты

`ReviewRow.test.tsx`:

1. `confident`-строка раскрывается кликом и показывает кандидатов + поиск.
2. У строки без `matched_code` в `candidates` (фонд-хит) рендерится
   синтетическая карточка; клик по ней вызывает `confirm` (не `pick`).
3. «Оставить без пары» доступна у `confident` и `matched_fund`;
   клик вызывает `reject`.
4. У синтетической карточки нет score, есть метка происхождения.
5. Регрессия на откат: после `pick` кандидата X на фонд-хите синтетическая
   карточка по-прежнему показывает исходную рекомендацию (снимок `matched_*`
   не изменился); клик по ней вызывает `confirm`, решение возвращается к
   исходной паре.

`reviewState.test.ts`:

6. `pick`/`reject` на `confident`-строке не меняет `progress()` —
   счётчик «проверено X из Y» инвариантен к правкам нетребующих решения строк.

`backend/tests/test_estimate_review.py` (закрепление инварианта, на который
опирается UI, — сейчас он нигде явно не проверяется):

7. В `test_pick_candidate_overridden` доассертить, что `matched_code` /
   `matched_article_id` в ответе PATCH остались исходными (AI-снимок не
   мутирует при `pick`). Аналогично один ассерт в reject-тесте.
