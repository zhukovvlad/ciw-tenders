# Дизайн: Золотой фонд решений (decision fund) — exact-match кэш подтверждённых сопоставлений перед RAG

**Дата:** 2026-06-30
**Статус:** дизайн утверждён (brainstorming, два раунда код-заземлённого ревью)
**Ветка:** `feat/decision-fund` (создаётся при реализации)

> **Суть.** Кэш «подтверждённое человеком решение по строке → статья», стоящий **перед** RAG-матчингом.
> Повторяющиеся строки сметы решаются мгновенно, детерминированно и без LLM из накопленных операторами
> решений; всё новое идёт в существующий RAG-путь как сегодня. Поглощает паузу
> [decision-cache-exploration] — она была заблокирована багом крошко-коллизии (Кейс D), который закрыт
> веткой `fix/breadcrumb-ancestor-collision` (PR #16): крошка стала устойчивым неколлизирующим ключом.

**Связано:** [TECH_DEBT.md](../../TECH_DEBT.md) «Качество матчинга» (Кейсы A–C, ради которых и нужен фонд),
[PIPELINE.md](../../PIPELINE.md), спека позиционного резолва
[2026-06-30-positional-hierarchy-resolution-design.md](2026-06-30-positional-hierarchy-resolution-design.md).

## 1. Проблема

RAG-матчинг (эмбеддинг крошки → pgvector top-k → `score > 0.90` confident, иначе LLM-арбитр) **заново
решает одни и те же строки в каждой смете**, включая те, где он стабильно ошибается. Измеренный пример
(golden, 809 узлов): узел-раздел `1.1.5 МОКАП` уходит в `1.99 Прочее` (needs_review 0.83), хотя в каталоге
есть `1.4 Мокап` — это retrieval-промах (Кейс B): различающий токен «МОКАП» тонет в общем многословном
предке, правильная статья даже не доходит до арбитра. Оператор правит это **в каждой смете заново**.

Но сметы компании **повторяемы** (переиспользуют один шаблон). Замер overlap нормализованных крошек
между реальными загрузками: 96–100% на переиспользовании шаблона (id12 «разделы 1-7» ↔ id3 образец = 96%),
с провалом до ~0% на сильно правленой ревизии. То есть бо́льшая часть строк новой сметы **байт-идентична**
уже виденным — их ответ можно не вычислять, а помнить.

**Ключевое наблюдение (почему ключ устойчив):** `embedding_input` строится из ИМЁН (предки + лист), **без
кодов разделов** и **с выброшенными org-предками** (орг-стрип). Значит ключ не зависит от нумерации (главный
источник хаоса — Кейс D) и от номера этапа: одна и та же работа даёт один ключ, в какой бы смете и под каким
бы кодом ни стояла.

---

## 2. Решение

**Фонд** — таблица `(нормализованный ключ-крошка) → (article_id)`, наполняемая из **подтверждённых
оператором** решений. На матчинге, **до арбитра**, по ключу узла ищем в фонде: единственный живой ответ →
применяем мгновенно (статус `matched_fund`, виден и переопределяем); промах/конфликт → строка идёт в RAG как
сегодня. Фонд — это **fast-path и институциональная память**, а не замена RAG: RAG остаётся для нового и
длинного хвоста.

### 2.1 Анти-утечка (train/test): бенчмарк ≠ источник

Бенчмарк (`benchmark_node`) — **независимый судья**, держится held-out. Засеять фонд из него и им же мерить =
100% по построению (leakage), метрика станет фиктивной. **Источник фонда — только ревью-решения оператора**
(`final_article_id` при `review_status ∈ {confirmed, overridden}`). Харнесс `eval_matching` гоняет с
**выключенным** фондом (`apply_fund=False`) → меряет чистый RAG.

## 3. Архитектура (Clean Architecture; направление `api → services → domain ← infrastructure`)

- **domain (чисто, без I/O):**
  - `normalize_cache_key(embedding_input: str) -> str` — детерминированная нормализация (регистр, пробелы,
    сепаратор) поверх уже-org-стрипнутой крошки.
  - **`CRUMB_DERIVATION_VERSION: int`** — **единый источник правды** версии крошко-деривации, в домене рядом с
    `build_embedding_input`. И промоушен (что пишем в `crumb_version`), и lookup (по какой версии ищем) читают
    **эту** константу — иначе рассинхрон «пишем X, ищем Y» молча уведёт весь фонд мимо под видом холодного
    старта. Меняем логику крошки (`build_embedding_input`/org-стрип/резолв предков) → бампаем константу.
    Версия — отдельный вход/колонка, НЕ внутри строки ключа.
  - `resolve_fund_decision(live_article_ids: Sequence[int]) -> int | None` — guard «единственный ответ»:
    ровно один → он; ноль или ≥2 → `None`. Принимает **только живые** id (фильтрация — в инфраструктуре).
  - сущность `FundDecision(article_id: int)` (+ метаданные по мере надобности).
- **domain/ports:** `DecisionFundRepository` — `lookup(keys, crumb_version) -> dict[key, list[FundHit]]`
  (живые попадания строго для переданной версии; **`crumb_version` — явный вход**, значение =
  `CRUMB_DERIVATION_VERSION`; `FundHit=(article_id, code, name)` — код/имя из JOIN к каталогу, чтобы не было
  N+1 `get_by_id` на хит; домен `resolve_fund_decision` всё равно решает по id), `promote(estimate_id)`,
  `set_reference(estimate_id, value: bool)`, `rebuild()`, `clear()`. Абстракция, без SQL.
- **infrastructure/db:** `SqlAlchemyDecisionFundRepository` + таблица `decision_fund` + флаг
  `estimates.is_reference` + Alembic-ревизия (вручную, как `0001`).
- **services:** в `EstimateMatchingService` — стадия `_apply_fund(estimate_id)` **перед** `_match_nodes`;
  промоушен — отдельный метод/сервис, дёргается тумблером. Выключатель `apply_fund: bool` через фабрику
  `build_estimate_matching_service(...)`.
- **api:** эндпоинт тумблера (двусторонний: ON → `is_reference=true` + промоушен; OFF → `is_reference=false`),
  админ-«пересобрать фонд», значение `matched_fund` в DTO-снимке строки.
- **frontend:** тумблер на этапе ревью/«готово»; бейдж «из фонда» в таблице строк.

## 4. Данные

### 4.1 Таблица `decision_fund`

| колонка | примечание |
|---|---|
| `id` PK | |
| `cache_key_hash CHAR(64)` | `sha256(normalize_cache_key(...))` — в unique-индекс (TEXT-ключ в btree-unique = мина по длине строки) |
| `cache_key TEXT` | сырой нормализованный ключ — **только для дебага**, не уникальный |
| `crumb_version SMALLINT` | версия крошко-деривации |
| `article_id INTEGER` | **БЕЗ FK** — зеркалит `estimate_rows.matched_article_id` (models.py:114); живость проверяется в lookup |
| `votes INTEGER` | **тупой неидемпотентный счётчик касаний**, инертен в v1 (guard его НЕ читает). Реальная семантика голосов (дедуп по `source_row_id`) — в D, через side-таблицу `fund_entry_sources(fund_id, source_row_id) UNIQUE`, `votes = COUNT`. В v1 не обещаем идемпотентности votes. |
| `origin TEXT` | `human_review` / `fund_derived` — инертен в v1 (всегда `human_review`), форвард-совместимость с D |
| `source_estimate_id INTEGER`, `source_row_id INTEGER` | аудит «последнего источника» (НЕ полный набор — для голосов D нужна side-таблица) |
| `created_at`, `updated_at` | |
| UNIQUE `(cache_key_hash, crumb_version, article_id)` | upsert: `ON CONFLICT … DO UPDATE SET votes = votes + 1, source_estimate_id = excluded.source_estimate_id, source_row_id = excluded.source_row_id, updated_at = now()` (иначе «последний источник» фактически хранил бы первый) |

**Guard «единственный ответ» считает различные `article_id` (строки), НЕ сумму `votes`.**

**Якорь = `article_id`** (PK `template_articles`), не `article_code`: коды мутабельны (импорт умеет recode),
id стабилен. Без FK — потому что `apply_plan` (import_repository.py:38) делает жёсткий
`DELETE FROM template_articles WHERE id IN (...)`; FK с RESTRICT уронил бы каталожный импорт. Снимки в
`estimate_rows` уже используют голый Integer ровно по этой причине.

### 4.2 Durable-набор источников

`estimates.is_reference BOOLEAN NOT NULL DEFAULT false`. Тумблер **двусторонний**: ON = `is_reference=true` +
промоушен; **OFF = `is_reference=false`** (снять смету из набора источников). Флаг делает эталонные сметы
**долгоживущими** (их строки нужны для rebuild и для пересборки при bump `crumb_version`). Без него
админ-rebuild и bump версии — пустышки (нечем перегенерировать). OFF-путь делает фонд **операбельным**:
ошибочно/конфликтно запромоученную смету снимаем (`set_reference(id, false)`) и `rebuild` перестраивает фонд
**без её вклада** — без ручной DB-хирургии (см. §6, §9).

### 4.3 Снимок фонд-хита в `estimate_rows`

`status='matched_fund'` (новое значение `EstimateRowStatus`), `matched_article_id` = решение фонда,
`matched_code`/`matched_name` = **текущие** код/имя статьи из каталога на момент применения (apply-time,
тем же lookup-JOIN), `candidates=NULL`, `score=NULL`, `review_status='unreviewed'`.

> **Снимок заморожен, как любой AI-снимок.** Для **новых** хитов rename/recode безвредны (тянем свежие
> code/name apply-time). Но уже-применённые `matched_fund`-строки в **старых** сметах несут замороженные
> `matched_code/name` — ретроактивного refresh снимок-модель не даёт (как и для `confident`/`needs_review`).

### 4.4 Совместимость со снимок-моделью (сверено по коду)

- `matched_fund ∉ fetch_matchable_nodes` (`status IN (pending,error,no_match)`) → арбитр его не берёт; статус —
  единственный рычаг скипа, независим от эмбеддинга.
- `matched_fund ∉ count_unfinished_nodes` (`status=='pending'`) → не триггерит `PARTIAL_ERROR`.
- `save_node_classifications` пишет только в `pending/excluded/error/no_match` → `matched_fund` идемпотентен
  на ре-прогоне.
- Ревью-кнопки работают: `_confirm` требует `matched_article_id` (у фонд-хита есть); `_pick`/ручной подбор
  при пустых кандидатах идёт через `get_by_id` (сверено по `estimate_review_service`).
- DTO: `candidates` уже `[]`-able, `score` уже `number|null` — снимок без них валиден. Фронт добавляет
  `matched_fund` в `MatchStatus` + бейдж «из фонда».

## 5. Поток

### 5.1 Промоушен (тумблер «в фонд»)

Authz: тумблер — владелец сметы или админ; админ-«пересобрать фонд» (§6) — только админ. Один commit на смету
(зеркало `save_node_classifications`). Тумблер двусторонний: ON → `is_reference=true` + промоушен; OFF →
`is_reference=false` (снять из набора источников; вклад исчезает после `rebuild`, см. §6/§9).

**Предикат строк (Q-promote — решение: только явно человеческие):**
`review_status ∈ {confirmed, overridden}` **И НЕ** (`status='matched_fund'` И `review_status='confirmed'`).

- Берём `confident`-unreviewed строки в фонд **НЕ** — это догадки AI (включая confident-but-wrong, напр.
  «Мобилизация 0.9161 → confident, арбитр не звался», TECH_DEBT.md:74-80), их кэширование размножило бы
  ошибку детерминированно. Тиер confident с пониженным весом — отложен в D.
- **Анти-накрутка (Q-loop):** фонд-хит, просто `confirmed` оператором → НЕ промоутим (новой информации нет).
  Фонд-хит, `overridden` оператором → промоутим выбранную статью как `human_review` (реальная правка).
- По каждой подходящей строке: `(sha256(normalize_cache_key(embedding_input)), final_article_id)` → upsert.

> **Одношот.** Тумблер промоутит состояние ревью **на момент клика**. Строки, подтверждённые позже, попадут
> в фонд только при повторном тумблере/rebuild. UX-заметка: жать «в фонд» по завершении ревью, не в середине.

### 5.2 Lookup-стадия `_apply_fund(estimate_id)` — ПЕРЕД `_match_nodes`, ПОСЛЕ `_embed_nodes`

- Кандидаты = **ВСЕ** `status='pending'`, `review_status='unreviewed'` узлы (НЕ «pending+вектор»: ключ —
  крошка, вектор не нужен; хит **спасает** узел с транзиентно отвалившимся эмбеддингом мимо `PARTIAL_ERROR`,
  промах безвекторного безвреден — он и так не в `fetch_matchable_nodes`).
- По ключу → `lookup(keys, CRUMB_DERIVATION_VERSION)` (строго текущая версия) → `resolve_fund_decision(живые
  article_id)`:
  - единственный живой ответ → пишем снимок `matched_fund` (`code/name` из того же lookup-JOIN, без N+1);
  - промах / конфликт (≥2 живых) / только мёртвые → оставляем `pending` → `fetch_matchable_nodes` берёт штатно
    в **том же прогоне** (вектор уже несёт).
- **Выключатель** `apply_fund: bool` через `build_estimate_matching_service(...)`: харнесс гоняет с `False`.

### 5.3 Условный гейт каталога

`matching_readiness()` в `match_estimate` проверяем **только если** после фонд-пасса остались не-фондовые
matchable-узлы. Полностью-фондовая смета не ловит спурьозный `DictionaryNotReadyError` (pending в каталоге →
retry/blocked на ровном месте).

### 5.4 Счётчик

Число фонд-хитов → в `_log_summary` (рядом с confident/needs_review/no_match/error/excluded).

## 6. Инвалидация (гибрид C)

| Событие каталога | Поведение фонда |
|---|---|
| **Добавление** статьи | фонд цел (старые ключи не трогаются) |
| **Переименование** (id тот же) | цел; новые хиты тянут свежее имя apply-time |
| **Перекодировка** `№` (id тот же) | цел (код для фонда не важен) |
| **Удаление** | lookup-JOIN отсекает мёртвый id **лениво**, авто; конфликт с мёртвым плечом схлопывается в живое |
| **Split / merge / смысл-дрейф** | id может выжить со смещённым смыслом. **Автодетектора в v1 НЕТ** (авто-разбор split/merge явно отложен, §8). Бэкстопы: видимый статус «из фонда» (оператор переопределит) + админ-«пересобрать фонд». |

> Харнесс к фонду **не подключён** (гоняет `apply_fund=False`, в `decision_fund` не смотрит; его метрики
> `gold_not_in_catalog`/`article_renamed` сравнивают gold бенчмарка с каталогом — обслуживание бенчмарка, не
> фонда). Не считать его детектором фондового дрейфа.

- **Админ «пересобрать фонд»** (authz: админ): `clear()` → пройти сметы с `is_reference=true` → перепромоутить
  из их финальных решений, мёртвые `article_id` отбросить. Дёшево, если `crumb_version` не менялся (читаем
  сохранённый `embedding_input` эталонных строк). **Снятая смета** (`is_reference=false`) в обход не попадает →
  rebuild = реальное удаление её вклада (исправление ошибочного промоушена без DB-хирургии).
  **Граница:** rebuild детерминирован над теми же `is_reference`-входами → конфликт, рождённый расхождением
  ДВУХ эталонных смет, bare-rebuild **воспроизводит** (ключ снова молчит). Снять такой конфликт = снять одну из
  смет (`set_reference false`) + rebuild, либо D (веса). См. §9.
- **Bump `crumb_version`** (сменили `build_embedding_input`/org-стрип/резолв предков): старые ключи мажут мимо
  на новой версии — **безопасно** (неверное не отдаём, просто холоднее до пересборки). Полная пересборка
  требует **переобработки** эталонных смет под новую крошко-логику (ре-прогон пайплайна, платно) — редко,
  админская/офлайн-операция.

## 7. Замер

- **«Доля узлов → `matched_fund`»** — честно как «сколько узлов фонд закрыл мимо арбитра», но это **верхняя
  граница** экономии арбитра, НЕ сама экономия: часть фонд-хитов в RAG ушла бы в `confident` (арбитр и так не
  звался) → для них экономия LLM = 0.
- **Реальная экономия LLM-вызовов** меряется **парным прогоном** одной сметы `apply_fund=False` vs `True` со
  счётом вызовов арбитра (разница = сэкономленные вызовы).
- **НЕ** экономия эмбеддинга — он платится всегда (фонд-хиты эмбеддинг не скипают).
- **Held-out целостность:** `eval_matching` с `apply_fund=False` меряет чистый RAG против бенчмарка, без
  утечки. Ценность фонда меряется отдельно на реальной смете (число хитов + выборочная проверка через
  видимый статус «из фонда»). На бенчмарк фонд не влияет by design.

## 8. Границы скоупа

**Входит в v1:** таблица `decision_fund` + `normalize_cache_key`/`resolve_fund_decision`; `is_reference` +
промоушен по тумблеру (предикат confirmed/overridden, анти-накрутка); `_apply_fund` перед `_match_nodes` со
статусом `matched_fund`; guard «единственный ответ»; условный гейт; выключатель для харнесса; инвалидация
гибрид-C (ленивый existence-JOIN + админ-rebuild + версия крошки); счётчик в summary; DTO/фронт-бейдж.

**НЕ входит (следующие итерации, интерфейсы заложены):**
- **D — голоса/консенсус/веса доверия:** side-таблица `fund_entry_sources` (дедуп по `source_row_id`),
  `votes=COUNT`, тиер `confident`-unreviewed с пониженным весом, направленное разрешение override
  (last-human-wins по свежести) вместо глушения.
- **E — acceptable-sets** (несколько приемлемых статей на узел) + правка метрики харнесса
  (`expected_article_code` single → set; top-1 hit = `chosen ∈ set`).
- **Негативные записи** (`key → no_article` из `REJECTED`) — см. §9.
- **Авто-разбор split/merge** каталога; bulk-импорт готовых смет как разовый seed.

**Не трогаем:** RAG (эмбеддинг/порог 0.90/арбитр), каталог, бенчмарк-судью (held-out, в фонд не идёт),
org-стрип/классификатор/`is_excluded`, схему `template_articles`.

## 9. Разрешённые открытые вопросы

- **Q-promote → только `confirmed`/`overridden`** (чистые человеческие решения). Высокоценное (исправленные
  needs_review, напр. МОКАП→1.4) ловим; малоценное-и-рисковое (`confident`-AI, документированный
  confident-but-wrong) пропускаем. Зависимость: выход зависит от того, что оператор реально жмёт confirm, а не
  только флипает тумблер сметы — UX, не блокер.
- **Q-loop → анти-накрутка** (фонд-хит+confirmed не промоутим; фонд-хит+overridden → как `human_review`).
- **Override фонд-хита → (а) глушим ключ** (новое плечо B + старое A → конфликт → молчим → RAG), а не
  заменяем. Удалять A (тоже человеческое решение из другой сметы) хуже, чем симметрично заглушить.
  **B сохраняется как данные** → слой голосов D разведёт направленно по свежести/весу.
  **Обратимость точно:** bare-rebuild конфликт НЕ снимает (детерминирован над теми же источниками,
  воспроизводит его) — снять можно только **изменением/снятием источника** (`set_reference false` на одну из
  расходящихся смет + rebuild) **или D** (веса). Не «обратимо через rebuild». Пометка в UX/девлоге:
  «переопределение фондового ответа отключает фонд для этого паттерна до снятия источника или пересборки с
  весами».
- **Q-rejected → отложить** (позитив-онли). Сильнейший довод: негативная запись протухает ровно на
  **добавлении** статьи, а добавление = «фонд цел, ключи не трогаются» → негатив молча застаревал бы. Позитив +
  ленивый JOIN на delete — чистая асимметрия в пользу отложения. Негатив + ведение staleness — в D.
- **Q-unreference → OFF снимает только флаг, вклад уходит через `rebuild()`, НЕ точечно.**
  `unreference(estimate_id)` — это ровно `set_reference(estimate_id, False)`; строки, уже записанные этой
  сметой в `decision_fund`, остаются активными до ближайшего `rebuild()`. Точечное удаление «вклада одной
  сметы» (delete по `source_estimate_id`) несостоятельно по трём причинам одновременно: (1) `source_*` —
  last-writer-only (§4.1/§4.2, durable-набор источников хранит только последнего вкладчика на ключ, не
  историю), значит нет гарантии, что удаляемая строка вообще принадлежит именно этой смете, а не более
  свежему вкладчику; (2) один ключ (`cache_key_hash`, `crumb_version`, `article_id`) законно разделяют
  несколько эталонных смет — delete снёс бы чужие подтверждения; (3) `votes` в v1 — счётчик без списка
  вкладчиков (D откладывает `fund_entry_sources`), поэтому нет способа проверить «остались ли ещё
  подтверждения» без полного пересчёта. Корректная и единственная поддерживаемая инвалидация — снять флаг
  (мгновенно, обратимо: `set_reference` туда-обратно) **плюс** `rebuild()` (детерминированно пересобирает
  фонд из оставшихся `is_reference=true` смет; см. §6 «Админ «пересобрать фонд»»). До `rebuild()` фонд
  временно шире факта — это принятый компромисс v1 (см. TECH_DEBT.md: авто-rebuild на OFF и side-таблица
  `fund_entry_sources` для точечной инвалидации — обе альтернативы отложены, не реализованы).

## 10. Тесты (TDD)

- **Домен (чисто):** `normalize_cache_key` — регистр/пробелы/сепаратор стабильны; разные `crumb_version` дают
  разные ключи. `resolve_fund_decision` — 1 живой → он; 0 → None; ≥2 → None.
- **Инфра:** lookup-JOIN отсекает мёртвый `article_id`; конфликт A(мёртв)+B(жив) → B.
- **Сервис:**
  - `_apply_fund`: single-answer хит на pending → снимок `matched_fund` (candidates=None, score=None,
    review=unreviewed); промах → pending; **embed-fail pending без вектора + хит → спасён**.
  - идемпотентность `matched_fund` на ре-прогоне (`save_node_classifications` не трогает).
  - **override фонд-хита → 2-е живое плечо по ключу → следующий `_apply_fund` по ключу = промах (RAG), не
    замена** (лочит выбор (а)).
  - **bump `crumb_version` → все старые ключи мажут мимо (холодный фонд)** (лочит семантику инвалидации); и
    обратное: promote(X) + lookup(X) при общей `CRUMB_DERIVATION_VERSION` → хит (лочит единый источник версии).
  - un-reference (`set_reference(id, false)`) + rebuild → вклад снятой сметы исчезает (лочит OFF-путь/обратимость).
  - условный гейт: полностью-фондовая смета + pending в каталоге → нет `DictionaryNotReadyError`.
  - `apply_fund=False` → фонд не трогается (чистый RAG).
  - предикат промоушена: confirmed/overridden промоутят; **фонд-хит+confirmed НЕ промоутится**;
    фонд-хит+overridden → выбранная статья как `human_review`.
  - счётчик хитов в `_log_summary`.
- **API/DTO/фронт:** ответ с `matched_fund`-строкой сериализуется (без кандидатов/скора, бейдж «из фонда»);
  тумблер двусторонний — ON ставит `is_reference=true` + промоутит, OFF снимает (authz владелец/админ);
  админ-rebuild (authz админ) чистит и перепромоучивает из `is_reference=true`-смет.

## 12. Амендменты 2026-07-02 (пост-мерж ревью PR #17)

Глубокое ревью реализации нашло дефекты, часть которых требует **изменить решения этой спеки**.
План реализации: [2026-07-02-decision-fund-review-fixes.md](../plans/2026-07-02-decision-fund-review-fixes.md).

### 12.1 Экспорт: `matched_fund` попадает в ячейку; формат «(код) Название» (меняет SP3 §5)

**Дефект.** Матрица SP3 не знала `matched_fund`: фонд-строка (`review_status='unreviewed'`, авто-confirm
только на фронте) проваливалась в «пусто» — экспорт молча терял ровно те строки, которые закрыл фонд.

**Решение.** Строка матрицы: `unreviewed + (confident | matched_fund)` → статья из снимка. Плюс
продуктовое изменение формата ячейки (просьба оператора): вместо голого кода — **`(код) Название`**
(`final_*` для confirmed/overridden, `matched_*` для авто-принятых). Без имени — голый код; без кода — пусто.

### 12.2 `_apply_fund` — ДО `_embed_nodes` (меняет §5.2 и §7)

**Было:** «ПЕРЕД `_match_nodes`, ПОСЛЕ `_embed_nodes`»; §7 фиксировал «эмбеддинг платится всегда».
**Дефект:** на повторной смете с высоким покрытием фонд-хиты всё равно оплачивают эмбеддинг-вызовы
OpenRouter + записи векторов, которые никогда не читаются — главная экономия кэша терялась.

**Стало:** порядок `classify → _apply_fund → embed → gate → match`. Свойства §5.2 сохранены:
- «хит спасает узел с отвалившимся эмбеддингом» — сохраняется тривиально (хиту вектор не нужен вообще);
- «промах идёт в RAG в том же прогоне» — сохраняется (промахи эмбеддятся сразу после фонд-пасса);
- «embed до гейта» (эмбеддинг остающихся узлов не впустую при неготовом каталоге) — сохраняется:
  гейт по-прежнему стоит после `_embed_nodes`;
- `fetch_unembedded_nodes` дополнительно исключает `matched_fund` (раньше — только `excluded`).
§7: пункт «НЕ экономия эмбеддинга» аннулирован — теперь фонд экономит и эмбеддинг фонд-хитов.

### 12.3 Ре-матчинг переразрешает `matched_fund` (меняет §4.4)

**Было:** `save_node_classifications` не трогает `matched_fund` («идемпотентен на ре-прогоне»).
**Дефект:** отравленный фонд неизлечим: после снятия плохой записи (unreference + rebuild) повторный
`POST /match` не пере-разрешал уже-проштампованные строки — неверная статья оставалась навсегда.

**Стало:** гард классификации дополнительно сбрасывает `matched_fund` **строго при
`review_status='unreviewed'`** → строка возвращается в `pending` и в том же прогоне переразрешается
текущим фондом (фонд не менялся → тот же снимок, исход идемпотентен) либо уходит в RAG (запись снята).
Ревью-решения оператора (confirmed/overridden) неприкосновенны, как раньше.

### 12.4 UI: фонд-хит реально переопределяем (закрывает разрыв §2/§6/§9)

**Дефект:** спека объявляла «виден, переопределяем» и строила на этом бэкстоп дрейфа (§6) и глушение
ключа через override (§9), но UI делал `matched_fund`-строку некликабельной — весь контур коррекции
был недостижим из продукта.

**Решение:** строка `matched_fund` раскрывается (ручной поиск → override), но НЕ требует решения
(без warning-рамки, прогресс не блокирует). `statusLabel` учитывает решение оператора: нетронутый
хит → «Из фонда», overridden → «Ручной выбор», rejected → «Нет совпадения».

### 12.5 Votes: повторный промоушен того же источника не накручивает (уточняет §4.1)

§4.1 оставлял votes «тупым неидемпотентным счётчиком». Уточнение (дёшево, строго лучше):
`ON CONFLICT ... votes = CASE WHEN source_estimate_id = excluded.source_estimate_id THEN votes
ELSE votes + 1 END` — тумблер ON→OFF→ON одной сметы больше не инфлирует консенсус. Полная семантика
голосов (side-таблица источников) — по-прежнему в D. Чередование двух смет по одному ключу всё ещё
может накручивать — принятый остаток v1.

### 12.6 Прочие дефекты ревью (спеку не меняют, фиксируются планом)

- **Дедуп промоушена:** повторяющаяся работа в одной смете = один conflict-ключ в батче;
  без дедупа Postgres кидает `CardinalityViolation` (500 на тумблере; rebuild умирал после `clear()`,
  оставляя фонд пустым). Дедуп по `(hash, version, article_id)` в `promote()` + честный фейк.
- **`promoted=0` → подсказка в UI:** тумблер молча отщёлкивался назад; теперь DoneScreen читает
  `promoted` и объясняет тостом («нет подтверждённых строк — пройдите ревью»).
- **Bulk-запись фонд-хитов:** `save_fund_hit` per-row commit → `save_fund_hits(hits)` одним
  commit-ом (зеркало `save_node_classifications»; N хитов ≠ N round-trip к Neon).
- Мелочи: `SqlAlchemyDecisionFundRepository` наследует порт-ABC; перекрёстные комментарии
  `CRUMB_DERIVATION_VERSION` ↔ `normalize_cache_key`; лёгкая проверка владения в `toggle_reference`
  (без загрузки строк с векторами); актуализация CLAUDE.md (таблица `decision_fund`, стадия фонда).
