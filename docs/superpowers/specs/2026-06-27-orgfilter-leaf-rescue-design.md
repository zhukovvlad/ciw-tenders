# Org-фильтр: спасение work-листьев под work-предком (FN-фикс) — Design

**Дата:** 2026-06-27
**Статус:** дизайн на ревью
**Связано:** [eval-харнесс](2026-06-26-matching-eval-harness-design.md) (измеритель), `docs/TECH_DEBT.md` 🔴-пункт «каркас-ЛИСТ под работным предком».

## Цель

Устранить «молчаливый пропуск работы»: лексический орг-фильтр исключает листовые узлы вида
«Корпус 8» / «Этап 1», сидящие под work-предком, хотя это работа, разбитая по корпусам/этапам.
Измерено харнессом на бенчмарке id=1: **Группа A FN=30**.

## Корень проблемы

`classify_lexical` (classification.py) смотрит **только на имя узла**: «Корпус 8» = оргтокен +
нет работной головы (цифры не ловятся `_TOKEN_RE`) → `ORG` → `excluded`, причём без LLM
(короткое замыкание). Контекст предков (`build_embedding_input` его учитывает) применяется
уже ПОСЛЕ классификации. Часть FN (узлы с работной головой, напр. «Этап 1 Индивидуальное
изготовление…») уходит в LLM, и LLM тоже ошибочно метит ORG.

Все 30 FN на бенчмарке — это узлы «Корпус N»/«Этап N» под work-предком (родитель kept=True,
реальная статья), причём gold у них = статья родителя.

## Правило (что меняем)

**Структурный override ПОСЛЕ классификации** (лексика + LLM не трогаем): узел, классифицированный
`ORG`, **оставляем** (`excluded=False`) тогда и только тогда, когда одновременно:

1. он **лист** (код-based: его код не является ничьим родительским кодом), и
2. у него есть **non-org предок** (`has_non_org_ancestor` — любой предок с классом ≠ `ORG`;
   эквивалентно «крошка непуста после выброса ORG»).

Иначе — `excluded`, как сейчас. Override применяется поверх **любого** ORG-вердикта (лексика ИЛИ
LLM): спасаемые узлы могут приходить из обоих источников, и асимметрия (FN дорог, ложно-оставленный
лист дёшев) оправдывает переопределение LLM.

Крошка спасённого узла строится из **work-контекста предков с выбросом собственного org-токена**
(своё имя не добавляется в `embedding_input`, ровно как выбрасываются ORG-предки). `has_non_org_ancestor`
гарантирует непустую крошку. Узел эмбедится по чистому work-контексту и матчится сам (без жёсткого
наследования статьи родителя).

### Почему именно «лист» и «non-org предок»

- **Лист** отделяет глубокие корпус-работы (спасаем) от структурных этап-разделителей с детьми
  (исключаем). На бенчмарке все 25 structural — это `3.1 «1 Этап ЖК»`, `4.1 «1 Этап ЖК»`… с детьми →
  остаются excluded.
- **Non-org предок** (а не `WORK`-предок): `UNSURE` — валидный финальный класс (LLM-фолбэк «сбой →
  весь батч UNSURE, никогда ORG»), трактуется как keep и наполняет крошку. Условие «есть `WORK`-предок»
  пропустило бы узел с UNSURE-предком → тихий FN. «Есть non-org предок» совпадает с «крошка непуста
  после выброса ORG» → ключевое свойство (kept ⟹ крошка непуста) держится по построению.

## Размещение (Подход 1 — сервисный слой)

`EstimateMatchingService._classify_nodes` (estimate_matching_service.py) — там доступно дерево
(`name_by_code`, `_ancestor_codes`, список узлов в порядке `source_index`). `classify_lexical` и
LLM-классификатор остаются чистыми/нетронутыми. Чистая логику решения выносим тестируемой функцией.

### A1. `classification.py` — `build_embedding_input` выбрасывает собственный org-токен

```python
def build_embedding_input(
    self_name, ancestors, *, self_class=WorkClass.WORK, separator=". ", collapse_repeats=True
):
    parts = [_normalize_ws(name) for name, cls in ancestors if cls is not WorkClass.ORG]
    if self_class is not WorkClass.ORG:          # своё имя — только если узел сам не ORG
        parts.append(_normalize_ws(self_name))
    if collapse_repeats:
        parts = _collapse_consecutive(parts)
    return separator.join(parts)
```
Дефолт `self_class=WORK` → существующие вызовы не меняются.

### A2. `classification.py` — чистая функция решения

```python
def is_excluded(own_class: WorkClass, *, is_leaf: bool, has_non_org_ancestor: bool) -> bool:
    """ORG исключаем, КРОМЕ листа с non-org предком (работа, разбитая по корпусам/этапам)."""
    if own_class is not WorkClass.ORG:
        return False
    return not (is_leaf and has_non_org_ancestor)
```

### A3. `_classify_nodes` — override в проходе 2

```python
parent_codes = {p for n in nodes if (p := self._parent_of(n.code))}  # код-based листовость
...
for n in nodes:
    # ancestors — ПОЛНЫЙ список (name, cls) по ВСЕМ предкам, БЕЗ предварительной фильтрации ORG
    # (как строится сейчас в проходе 2). Несущее: и has_non_org_anc, и build_embedding_input
    # фильтруют ORG ИЗНУТРИ по этому же списку — отсюда эквивалентность инварианта 2 ниже.
    ancestors = [(name_by_code[a], cls_by_id[repr_id_by_code[a]])
                 for a in self._ancestor_codes(n.code) if a in name_by_code]
    own = cls_by_id[n.id]
    is_leaf = n.code not in parent_codes
    has_non_org_anc = any(cls is not WorkClass.ORG for _, cls in ancestors)
    excluded = is_excluded(own, is_leaf=is_leaf, has_non_org_ancestor=has_non_org_anc)
    crumb = build_embedding_input(n.name, ancestors, self_class=own)
    # Инвариант 2 (защитный): kept-узел ОБЯЗАН иметь непустую крошку. Срабатывает НЕ на логике
    # _classify_nodes (is_excluded для ORG-без-non-org-предка вернёт True), а ловит будущий
    # РАССИНХРОН между is_excluded и build_embedding_input (поменяли одно, забыли другое).
    if not excluded and not crumb:
        logger.error("kept-узел с пустой крошкой: id=%s code=%s class=%s", n.id, n.code, own)
        raise AssertionError(f"kept node with empty crumb: {n.id} {n.code} {own}")
    results.append(NodeClassification(node_id=n.id, excluded=excluded, embedding_input=crumb))
```
`has_non_org_anc` считается по базовой классификации (до override); override меняет только листья
(у листа нет потомков) → каскада нет, проход однопроходный (проверено на данных, см. ниже).

## Листовость: код-based (D3), позиционная отвергнута

Решение принято **по данным**, не по рассуждению.

- **Код-based** (`код ∉ родительские коды`): консистентна с код-сегментной иерархией всего пайплайна;
  на бенчмарке корректна для всех ORG-узлов, которые правило гейтит.
- **Позиционная** (`source_index`+`depth`, лист ⟺ у следующего depth ≤ текущего) **отвергнута**:
  сверка двух листовостей на бенчмарке дала **16 расхождений**, т.к. смета — **не строгий DFS-преордер**
  (5 разрывов порядка: `3.2.4→3.1.4.1`, `5.2→5.1.1`, `8.2.3→8.1.3.1`, `10.1.2→10.2.1.1`,
  `11.1.3.1→11.3.1.1.1` — следующая строка глубже, но из другого поддерева). 15 из 16 расхождений —
  на work-узлах (правило не гейтит); единственная «победа» позиционной на правило-релевантном `5.2`
  **случайна** (приняла `5.1.1` за ребёнка `5.2`). Позиционная ненадёжна на реальных данных.
- Слабость код-based под дублями (11 расхождений) — **целиком на work-узлах** → ноль влияния на исход
  правила. Под дублями код-based неоднозначна **по построению** (если дублированный код — лист под
  одним родителем и не-лист под другим, оба получат одинаковый флаг), но на ORG-узлах это не стреляет.

## Инварианты

1. **Синхронизация с матрицей Группы A (Issue 1).** Спасаемый org-лист в gold размечается `matchable`
   (его статья = инherited работа), поэтому `matchable + kept = TP`, не FP. `structural` =
   org-узел **без** work-контекста ИЛИ **не-лист**. **Оборотная сторона:** если разметчик на новом
   объекте пометит спасаемый корпус-лист как `structural`, правило даст FP — это сигнал **рассинхрона
   разметки с правилом**, не баг правила. Матрицу менять не нужно.
2. **kept ⟹ крошка непуста.** `is_excluded=False` для ORG возвращается только при
   `has_non_org_ancestor=True` ⟹ после выброса ORG крошка непуста (эквивалентность держится, т.к.
   `has_non_org_anc` и `build_embedding_input` фильтруют ORG по одному и тому же полному `ancestors`).
   Прочие kept-узлы (WORK/UNSURE-корень depth 1) дают крошку = своё имя. **Защитный ассерт проведён в
   код A3** — в цикле, прямо перед `results.append` (точка определена). Он НЕ ловит логику
   `_classify_nodes` (она по построению не может породить kept-узел с пустой крошкой), а защищает от
   будущего **рассинхрона** между `is_excluded` и `build_embedding_input` (изменили одну, забыли другую).
   Лог с (id, code, class) указывает место поломки.

## Допущения (явные, не доказанные инварианты)

- **D1 — этап-разделители непустые (с детьми).** Несущая нагрузка leaf-условия. **Проваливается на
  бенчмарке** на `5.2 «2 Этап БЦ»` — это бездетный этап-разделитель с work-предком (5), который правило
  ошибочно спасает → **FP=1**. Принято по асимметрии (см. ниже). **Встроенный датчик с порогом:** этот FP
  виден в харнессе как `structural + kept`. **Baseline на id=1: FP=1.** Интерпретация порога: FP Группы A
  должен примерно равняться числу **бездетных org-листьев с work-предком** в объекте (на id=1 это ровно
  один — `5.2`). Заметное превышение = либо рассинхрон разметки с правилом (Issue 1), либо D1 проседает
  системно (много бездетных этап-разделителей). То есть «бить тревогу» не на любом FP>0, а на FP, ощутимо
  выше числа таких листьев.
- **Зависимость от вердикта классификатора.** «FP=1 / FN=2» — результат **прогона id=1 с конкретным
  вердиктом классификатора**, не абсолютное свойство правила. Разводим явно:
  - **FP=1 на `5.2` — стабилен.** `5.2` — чистый каркас, классифицируется ORG **лексикой**
    (детерминированно, без LLM) → этот FP воспроизводится на каждом прогоне.
  - **Плавать может вклад смешанных узлов** (классифицируются LLM, недетерминизм): на другом прогоне
    какой-то смешанный узел может попасть/выпасть из ORG, сдвинув FP/FN на ±1. Это не регрессия, а та же
    LLM-зыбкость, что и во всей LLM-зависимой части Группы A. Сейчас в FP такие узлы не попали — в FP=1
    сидит только лексический `5.2`.

## Замер (DoD)

Ре-прогон `just eval-matching` по бенчмарку id=1 после фикса. Ожидаемая матрица Группы A
(полная симуляция правила, код-based лист, проверено на данных):

```
ДО:    TN=25  FP=0  FN=30  TP=753
ПОСЛЕ: TN=24  FP=1  FN=2   TP=781
```

- **FN 30→2**: спасены 28 узлов. Остаток 2 (`8.1.2.10.1 «Корпуса 1,2,3,4,5,6,7,9»`,
  `8.2.2.10.1 «Корпус 8»`) — промежуточные org-узлы с детьми; **доказанно дёшевы**: их статья 8.2.5
  полностью несётся всеми 19 детьми (14+5; kept, в основном confident), обёртка избыточна.
- **FP 0→1**: `5.2 «2 Этап БЦ»` (допущение D1).
- **TP 753→781**: прирост реальный, но среди 28 спасённых — **8 уникальных gold-статей** (кластеры
  навесов: 9.2×8, 20.2×7, 3.1×3, 12.4×3…). В отчёте замера давать **обе** цифры: спасённых узлов (28)
  и уникальных статей (8) — иначе «+28 TP» читается как 28 разных побед.
- **Каскад:** 0 спасаемых являются предком ORG-узла → однопроходная матрица точна (проверено).

Замер на **одном** объекте — это валидация, что правило делает задуманное, **не** что оно обобщается.
D1 и зависимость от вердикта — первые кандидаты на провал на втором объекте.

## Краевые случаи

1. **Этап-разделитель-лист** (`5.2`) → спасается как FP. Принято (D1), датчик = FP Группы A.
2. **Пустая крошка у kept-узла** → невозможна по построению; защитный ассерт ловит будущий баг.
3. **WORK/UNSURE с оргтокеном в своём имени** («работы корпуса 5») → `self_class ≠ ORG` → своё имя НЕ
   режется. Корректно.
4. **Дубли кодов** → листовость код-based (анализ неоднозначности под дублями — в §«Листовость», не
   дублируем; на ORG-узлах не стреляет); крошка/предки по `name_by_code`/`_ancestor_codes`
   (существующая код-сегментная модель, не трогаем).
5. **Org-узел со всеми предками org** → `has_non_org_ancestor=False` → excluded (крошка была бы пустой).
6. **Дефолт `self_class=WORK`** → обратная совместимость существующих вызовов.

## Тесты

**Юнит (`classification.py`, чистые):**
- `is_excluded`: ORG+leaf+non-org-предок→keep; ORG+не-лист→exclude; ORG+leaf+нет non-org-предка→exclude;
  WORK→keep; UNSURE→keep.
- `build_embedding_input`: `self_class=ORG`→своё имя выброшено; `self_class=WORK` (дефолт)→на месте;
  все предки ORG + self ORG → пустая строка.

**Сервис (`_classify_nodes`, фейки портов, без БД/AI):** мини-дерево — work-родитель → org-лист (ждём
kept + крошка = work-контекст, своё имя вне); верхний org-разделитель с детьми (excluded); org-лист
без non-org предка (excluded); UNSURE-предок → org-лист под ним спасается (регресс на `has_non_org`
vs `WORK`).

**Защитный инвариант пустой крошки (привязка к точке A3).** «Честный» kept-узел с пустой крошкой
`_classify_nodes` породить не может (его `is_excluded` для ORG-без-non-org-предка вернёт `True`) —
значит тест должен **индуцировать рассинхрон** `is_excluded`↔`build_embedding_input`, против которого
ассерт и стоит: например, monkeypatch `build_embedding_input` → `""` (или `is_excluded` → `False`) на
узле, и проверить, что цикл в `_classify_nodes` поднимает `AssertionError` с (id, code, class) прямо
перед `results.append`. Тест бьёт именно в эту строку, а не в недостижимое естественное состояние.

**Замер (харнесс, не юнит):** ре-прогон → матрица выше + число уникальных статей.

## Вне области (YAGNI)

- Различение `5.2` от корпус-листа (нет надёжного структурного признака; глубина — хрупкий прокси;
  LLM-проход недетерминирован ради 1 дешёвого FP).

- Восстановление 2 остаточных FN (доказанно дёшевы, работа ловится детьми).
