# Спецификация: загрузка справочника СМР и фоновый эмбеддинг

**Дата:** 2026-06-21
**Статус:** дизайн утверждён, готов к плану реализации

## Задача

Дать админу способ загрузить эталонный справочник статей СМР из Excel-файла,
санитизировать строки, построить из них дерево «раздел → подраздел → работа» в
`template_articles`, и наполнить векторные эмбеддинги в фоне отдельным воркером.

Это первый шаг RAG-ядра: пока речь только про **наполнение справочника**.
Сопоставление смет (матчинг) — отдельная задача и здесь не перерабатывается.

## Контекст и исходные данные

- Файл-шаблон ([temp/Шаблон.xlsx](../../../temp/Шаблон.xlsx)): один столбец, 362 строки
  формата `(КОД) Наименование`. Иерархия закодирована в самом коде:
  `(1.)` раздел → `(1.1.)` подраздел → `(1.4.1.)` лист. Глубина до 3 уровней
  (21 / 126 / 214). Встречается «грязь»: `(6.6 .)` (пробел внутри кода),
  `(1.99.)` / `(99.)` — служебные «Прочее».
- Проверено на реальном файле: после нормализации все 362 кода уникальны, ни одной
  строки-сироты (родитель отсутствует), пустых имён нет.
- Реальная схема БД (Alembic-ревизия `0001`): `template_articles` — это **дерево**
  через `parent_id` (self-FK, `ON DELETE CASCADE`), `article_code VARCHAR(64) UNIQUE`,
  `name TEXT`, `embedding VECTOR(768)`, `created_at` / `updated_at`. Колонки
  `section_name` в БД **нет**.
- 🔴 Тех-долг (см. [docs/TECH_DEBT.md](../../TECH_DEBT.md)): ORM-модель `TemplateArticleModel`,
  доменная сущность `TemplateArticle`, репозиторий, `article_service`, `anthropic_matcher`
  и DTO всё ещё на старой плоской модели `section_name`, расходясь с реальной схемой.
  Эта задача закрывает долг в части слоя данных.

## Ключевые решения (зафиксированы с заказчиком)

1. **Триггер загрузки** — admin API endpoint (`POST /api/articles/import`, multipart `.xlsx`).
2. **Эмбеддинг — в фоне**, отдельным процессом-воркером. Очередь = сама колонка
   `embedding IS NULL` (никаких Celery/Redis: против ограничений «без Docker, общая
   Windows-машина»).
3. **Повторная загрузка** — upsert по `article_code` (а не wipe & reload).
4. **`POST /articles`** (ручное создание одной статьи) — адаптируем под дерево, не удаляем.
5. **Порядок отображения** — сортировка по коду численно
   (`ORDER BY string_to_array(article_code, '.')::int[]`), отдельной колонки `sort_order`
   нет. Код уже задаёт правильную иерархическую позицию; ручная вставка статьи
   автоматически встаёт на место по своему коду.
6. **Единый путь векторизации документов** — никто не эмбеддит статьи синхронно. Любая
   запись статьи (импорт *и* ручной `POST`) кладёт `embedding_input` + `embedding = NULL`;
   заполняет вектор только воркер. Эмбеддинг *запроса* при матчинге остаётся синхронным
   (это другой путь, его не трогаем).

## Архитектура — две фазы

```
[admin] --upload .xlsx--> POST /api/articles/import   (синхронно, быстро)
                                  │
                                  ▼
                 TemplateParser  →  TemplateIngestService
              (парсинг+санитайз)     (дерево, upsert, инвалидация)
                                  │  пишет: name + embedding_input + embedding=NULL
                                  ▼
                          template_articles  ◄── очередь = (embedding IS NULL)
                                  ▲
                                  │  (асинхронно, отдельный процесс)
                     app.scripts.embed_worker  →  Embedder (Gemini, батчами)
```

Слои Clean Architecture сохраняются: сервисы зависят только от портов
(`ArticleRepository`, `Embedder`); внешние SDK — в `infrastructure/`.

## Модель данных

Дерево уже есть в ревизии `0001`. Добавляется **миграция `0002`**:

| колонка | тип | источник | назначение |
|---|---|---|---|
| `id` | `SERIAL PK` | 0001 | — |
| `parent_id` | `INTEGER FK self, ON DELETE CASCADE` | 0001 | иерархия |
| `article_code` | `VARCHAR(64) UNIQUE NOT NULL` | 0001 | нормализованный код, напр. `1.4.1` |
| `name` | `TEXT NOT NULL` | 0001 | санитизированное имя |
| `embedding` | `VECTOR(768)` | 0001 | `NULL` = «в очереди на эмбеддинг» |
| **`embedding_input`** | `TEXT NOT NULL` | **0002 (новое)** | обогащённый текст, которым делали/сделаем вектор |
| `created_at` / `updated_at` | `timestamptz` | 0001 | — |

`embedding_input` — ключ к корректной инвалидации эмбеддинга при upsert и одновременно
развязка воркера от дерева (воркер читает готовую строку, ему не нужна иерархия).

Миграция `0002` для совместимости с автогенерацией: добавить колонку `embedding_input`.
Поскольку таблица на практике пуста (ORM был сломан и в реальную схему не писал), колонка
заводится сразу `NOT NULL`. Если в окружении найдутся легаси-строки — бэкфилл значением из
`name` и `embedding = NULL` (переэмбеддятся воркером). ORM-фильтр для `Vector`/HNSW в
`alembic/env.py` добавить при первой автогенерации (см. тех-долг).

**Закрытие долга в слое данных:** `TemplateArticle` (entity), `TemplateArticleModel` (ORM),
`SqlAlchemyArticleRepository` приводятся к дереву. Поле `section_name` удаляется; появляются
`parent_id` и `embedding_input`.

Итоговая доменная сущность:

```
TemplateArticle:
    article_code: str
    name: str
    embedding_input: str
    parent_id: int | None = None
    id: int | None = None
    embedding: list[float] | None = None
```

## Компонент 1: `TemplateParser` (services)

Single Responsibility: распарсить и санитизировать файл-шаблон. Без БД и без эмбеддера.
Отдельный класс от `ExcelEstimateParser` (тот парсит сметы — другой формат).

Вход: `bytes` (.xlsx). Выход: список `ParsedTemplateRow` + список пропущенных строк.

```
ParsedTemplateRow:
    article_code: str       # нормализованный, напр. "1.4.1"
    name: str               # санитизированное имя
    parent_code: str | None # код родителя или None для корня
    embedding_input: str    # обогащённый текст
```

Правила обработки строки `(КОД) Наименование`:

- **Код:** regex `^\((.*?)\)\s*(.*)$`; из группы 1 убрать все пробелы и хвостовую точку
  → `(6.6 .)` → `6.6`, `(1.4.1.)` → `1.4.1`. Все сегменты кода обязаны быть числовыми
  (валидируется; нужно для численной сортировки по коду).
- **Имя:** `strip`, схлопнуть повторные пробелы в один, NBSP → пробел. Регистр и скобки
  внутри имени сохраняются (`Прочее (котлован)`).
- **parent_code:** код минус последний сегмент (`1.4.1` → `1.4`; `1` → `None`).
- **Обогащённый текст** (`embedding_input`): имена всех предков от корня + собственное имя,
  через `. `. Пример: `(1.4.1.) Мокап фасада` →
  `"Подготовительные работы, содержание площадки. Мокап. Мокап фасада"`
  (тот же стиль конкатенации, что использовался в прежнем `article_service`).

Обработка ошибок:

- Строка без скобок `(...)` или с пустым именем → не валит импорт, попадает в список
  «пропущено» (возвращается в отчёте).
- Нечисловой сегмент кода → строка в «пропущено».
- Дубликат кода в файле → ошибка импорта целиком (нарушит UNIQUE-констрейнт).

## Компонент 2: `TemplateIngestService` (services)

Зависит от `ArticleRepository`. **Не** зависит от `Embedder` (эмбеддинг — воркер).

`import_template(content: bytes) -> ImportReport`:

1. `TemplateParser.parse(content)` → строки + пропущенные.
2. Валидация целостности: каждый непустой `parent_code` должен присутствовать в наборе
   кодов файла (нет сирот). Иначе — ошибка импорта.
3. Загрузить из БД текущее состояние: `code → (id, embedding_input)`.
4. Вычислить дельту:
   - **insert** — коды, которых нет в БД;
   - **update** — коды, что есть в обоих (обновляем `name`, при необходимости `parent_id`);
   - **delete** — коды в БД, которых нет в файле.
5. **Инвалидация эмбеддинга:** для insert и для update, где новый `embedding_input` ≠
   хранимому, ставим `embedding = NULL`. У строк с неизменившимся `embedding_input` вектор
   сохраняется (экономим вызовы Gemini).
6. **Резолв `parent_id`** двухпроходно: сначала upsert всех строк (родители раньше детей по
   длине кода либо `parent_id = NULL` на первом проходе), затем второй проход — проставить
   `parent_id` по `article_code` родителя.
7. Всё в одной транзакции.
8. Вернуть отчёт:
   ```
   ImportReport: created, updated, deleted, unchanged, skipped[], pending_embeddings
   ```

Новые методы репозитория (расширение `ArticleRepository` или отдельные bulk-методы):
выборка `code → (id, embedding_input)`; bulk insert/update/delete; проставление `parent_id`.
Конкретный набор методов уточняется в плане реализации.

## Компонент 3: `embed_worker` (scripts)

`uv run python -m app.scripts.embed_worker [--once] [--batch-size N]`.
Переиспользует порт `Embedder` (Gemini) и сессию БД. Цикл:

1. Взять пачку строк `WHERE embedding IS NULL` (по умолчанию 100 — учёт лимитов Gemini
   на размер батча).
2. `embedder.embed_batch([embedding_input, ...])`.
3. Записать векторы по `id`, коммит пачкой.
4. Нет строк → при `--once` завершиться; иначе короткий sleep и повтор.

Свойства: без брокера; падение/рестарт безопасны — невекторизованные строки остаются
`IS NULL` и подберутся при следующем проходе.

## Компонент 4: API и сопутствующие правки

- **`POST /api/articles/import`** — `require_admin`, multipart `.xlsx`. Возвращает `ImportReport`.
  Ошибки парсинга/целостности → `400` с понятным сообщением.
- **`GET /api/articles`** — отдаёт дерево/список, упорядоченный по коду численно
  (`ORDER BY string_to_array(article_code, '.')::int[]`). DTO `ArticleOut`: `section_name`
  убирается, добавляются `article_code` / `parent_id`.
- **`POST /api/articles`** — адаптируется под дерево: принимает `article_code`, `name`,
  опциональный `parent_code`; вычисляет `embedding_input` (контекст предка из БД), пишет
  `embedding = NULL` (без синхронного эмбеддинга — заполнит воркер).
- **`DELETE /api/articles/{id}`** — без изменений (каскад по `parent_id` уже в схеме).
- **`anthropic_matcher`** — листинг кандидатов сейчас печатает `section_name`; меняем на
  `name` (контекст уже зашит в `embedding_input`). Полная переработка матчинга под дерево —
  отдельная задача (сметы).
- **`ArticleService`** теряет зависимость от `Embedder` (синхронного эмбеддинга больше нет).

## Тестирование

Юнит-тесты не ходят в реальную БД/AI — фейки портов ([tests/fakes.py](../../../backend/tests/fakes.py))
и `app.dependency_overrides`.

- `TemplateParser`: нормализация кодов (вкл. `(6.6 .)`), санитайз имён, parent_code,
  построение `embedding_input`, «грязные»/пустые строки → skipped, нечисловой код → skipped.
- `TemplateIngestService`: insert / update / delete / unchanged; инвалидация эмбеддинга при
  смене имени предка (потомки → `embedding = NULL`); валидация сирот; дубликат кода → ошибка.
- `embed_worker`: фейк-эмбеддер; строки `IS NULL` заполняются батчами; `--once` завершается.
- Эндпоинт: загрузка мини-`.xlsx` через `TestClient`, проверка отчёта и enforcement `admin`.

## Вне объёма

- Переработка алгоритма матчинга под дерево (контекст предков в поиске/арбитраже).
- Фронтенд-экран загрузки справочника и просмотра дерева.
- Полноценная очередь задач (Celery/Redis), мониторинг, ретраи с backoff.
- Колонка `sort_order` (порядок берём из кода).
```
