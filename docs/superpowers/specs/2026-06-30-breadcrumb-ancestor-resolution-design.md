**Связано:** [TECH_DEBT.md](../../TECH_DEBT.md) «Качество матчинга → Кейс D»

## 1. Проблема

Крошка узла сметы (`embedding_input`) собирается из имён предков: для узла с кодом
`6.4.1` предки — коды `6` и `6.4`, их имена клеятся в строку «Имя(6). Имя(6.4). Имя(6.4.1)».
Имя предка по коду берётся из словаря `код → имя`, заполняемого **первым вхождением** кода.

В многоэтапных сметах коды дублируются между этапами с разным смыслом. Пример (глава 6, «Фасады»):

| код | Этап 1 (ЖК) | Этап 2 (БЦ) |
|---|---|---|
| `6.4` | «навесной фасад **типового** этажа» | «навесной фасад **1** этажа» |
| `6.5` | «**модульный** фасад» | «навесной фасад **типового** этажа» |

Словарь «первого вхождения» отдаёт для `6.4` всегда «типового», для `6.5` всегда «модульный».
Поэтому строка `6.4.1` Этапа 2 (реально — «1 этажа») получает предка «типового» → крошка врёт →
вектор уезжает к неверной статье. Баг **тихий** (неверный матч без ошибки), бьёт по всем
многоэтапным сметам.

Корень: предки резолвятся через **глобальный словарь** (теряет позицию), а не по фактическому
положению узла в потоке строк.

## 2. Решение: nearest-preceding (ближайшее предшествующее вхождение)

Имя предка берём не из «первого вхождения за всю смету», а из **ближайшего вхождения того же
кода ВЫШЕ по документу** (макс. `source_index` < узла). Если предшествующего нет — откат к первому
вхождению.

Аналогия: чтобы понять раздел абзаца, человек смотрит на ближайший заголовок над абзацем, а не на
самый первый заголовок документа.

### Дифференциальный результат на golden-смете

41 узел получает исправленную крошку, **0 регрессий** (ни один не теряет контекст относительно
текущего поведения). Бенчмарк 2 («разделы 1-4») — 0 изменений.

### Достаточность code-prefix: верна для главы 6, НЕ везде (измерено)

**Глава 6 (фасады):** сверка nearest-preceding против outline-восстановления — этажный токен
восстановлен у всех коллизионных узлов, расхождений по этажу **0**. Причина: коды видов работ внутри
главы вкладываются корректно (`6.2.1` под `6.2`), хаотичны лишь коды-обёртки этапов, а они ORG →
срезаются. Здесь outline был бы избыточен.

**Но на ЭТОЙ ЖЕ смете code-prefix недостаточен вне главы 6 (измерено, бенчмарк 1):**

- **Глава 10 — инъекция контекста СНИЗУ (4 узла, 8 откат-резолвов):** `10.2.1.1/.2/.3` «Монтаж
  опор…» наружного освещения ЖК не имеют предшествующего `10.2` → откат тянет первое вхождение `10.2`
  из блока «2 Этап Офис» в ~113 строках НИЖЕ. Крошка получает «фасадные осветительные приборы 2 этап
  офис» — чужой рабочий термин чужого этапа.
- **Глава 11 — потеря рабочих предков (raised в ревью):** усечение кода даёт не тех предков, чем
  outline (напр. «Тип 11 Устройство газона» теряет «Озеленение. Покрытие по грунту»). Это **отдельный
  режим** (не откат), флаг `is_fallback` его не ловит.

**Важно: это НЕ регрессия.** Для узлов отката nearest-preceding берёт ровно то же первое вхождение,
что и текущий словарь — они были неверны и остаются неверными, **осознанно не чинятся в этом патче**
(см. §6). Патч остаётся строгим улучшением: +41 в главе 6, 0 регрессий.

## 3. Архитектура

### §3.1 Чистая функция в домене

В [domain/classification.py](../../../backend/app/domain/classification.py) (рядом с
`build_embedding_input`):

```python
def resolve_ancestor_indices(codes: Sequence[str]) -> list[list[tuple[int, bool]]]:
    """codes — коды узлов В ПОРЯДКЕ документа. Для каждой позиции i возвращает
    список (индекс_предка, is_fallback) root→parent. Код-предок разрешается в
    БЛИЖАЙШЕЕ предшествующее вхождение (макс. индекс < i, is_fallback=False); если
    предшествующего нет — первое вхождение (is_fallback=True; индекс может быть ≥ i,
    контекст тянется СНИЗУ — крошка для такого предка НЕНАДЁЖНА). Код-предок, не
    встречающийся вовсе, пропускается (как сейчас).

    Инвариант (тестируемый благодаря флагу): для предков с is_fallback=False индексы
    строго возрастают и < i; элементы is_fallback=True — единственное исключение из
    «< i». Этот же флаг — сигнал «крошка здесь ненадёжна» для пост-замерного среза."""
```

- Принимает **только упорядоченные коды**, возвращает **индексы + флаг** (не имена) → функция чистая,
  без ORM/имён/классов, тривиально тестируется.
- **Флаг `is_fallback` обязателен** (правка по ревью): на плоском списке индексов property-тест
  монотонности непроверяем — он либо упал бы на узлах отката-вперёд, либо проходил вхолостую. Флаг
  делает инвариант точным И даёт сигнал ненадёжности для пост-замера.

### §3.2 Где используется (scope A — оба слоя)

Все три заменяют резолв «первого вхождения по коду» на `resolve_ancestor_indices`, мапят индекс →
`(имя, класс)`. Крошка строится из **всех** предков (включая откатные) — это сохраняет паритет с
текущим поведением (без регрессии); `is_fallback` используется только тестами и пост-замером, не
меняет саму крошку.

1. **`_classify_nodes`** ([estimate_matching_service.py:154-158](../../../backend/app/services/estimate_matching_service.py#L154))
   — **авторитетный путь**: предки `(имя, класс)` для `build_embedding_input`. Чинит эмбеддинг/матчинг.
2. **`_ancestor_names`** ([estimate_matching_service.py:187](../../../backend/app/services/estimate_matching_service.py#L187))
   — контекст предков для LLM-классификатора.
3. **`estimate_parser`** ([estimate_parser.py:76-84](../../../backend/app/services/estimate_parser.py#L76))
   — крошка на парсинге (consistency; её перезатирает классификация, но два расходящихся способа
   строить дерево держать нельзя).

## 4. Тесты (TDD: red → green)

### Юнит на `resolve_ancestor_indices` (чистый, без БД/AI)

- **Фасад `6.4.1`:** `6.4` ЖК = «навесной типового», БЦ = «навесной 1 этажа» → две строки `6.4.1`
  дают **разных** предков (gold: ЖК→6.4.1, БЦ→6.3.1).
- **Фасад `6.5.1`:** `6.5` ЖК = «модульный», БЦ = «навесной типового» → `6.5.1` БЦ обязан дать
  «навесной типового», НЕ «модульный» (gold БЦ→6.4.1). Тестирует **превосходство над первым вхождением**.
- **Перегородки `5.1.1`:** заголовок секции во 2-м этапе не повторён → ближайший = первое вхождение
  (контекст сохранён).
- **Откат тянет контекст СНИЗУ (документированно-неверная крошка):** `10.2.1.1` «Монтаж опор» (гл.10,
  наружное освещение ЖК) не имеет предшествующего `10.2` → откат вставляет «фасадные осветительные
  приборы 2 этап Офис» из ~113 строк ниже, `is_fallback=True`. Тест фиксирует это как **сознательно
  НЕ чинимую** в этом патче неверную крошку, равную текущему поведению — НЕ «безобидный край».
- **Property — монотонность (через флаг):** для предков `is_fallback=False` индексы строго возрастают
  и `< i`; элементы `is_fallback=True` — единственное исключение.

### Сервис-тест на `_classify_nodes`

На узлах с коллизирующим кодом `embedding_input` двух строк выходит **разным** (сейчас — байт-в-байт
одинаков). Ловит интеграцию функции в авторитетный слой.

### Пост-замер (разово, не в CI)

`eval_matching` до/после на golden-смете → дельта **top-1 / top-3**. **Именованный срез:** 4 узла
отката-снизу (`10.2.1.x`, гл.10) — отдельно меряем, флипает ли инъекция чужого контекста их top-1.
**Метрика `is_leaf`** (§5) в том же diff. Раз OpenRouter живой — снимаем честную цифру.

## 5. Границы скоупа

**Не трогаем:** org-стрип, порог 0.90, LLM-арбитр, каталог, схему БД.

**Не делаем «корпус»:** иерархический матчинг, смену структурного сигнала — отдельный стратегический
разговор после пост-замера распределения промахов.

**`is_leaf` — фикс откладываем, но дисциплинированно.** `is_leaf`
([estimate_matching_service.py:160](../../../backend/app/services/estimate_matching_service.py#L160))
определяется через код-based `parent_codes` и болеет **тем же** механизмом коллизии. В этом патче:

1. **Характеризующий тест сейчас** — фиксирует текущее поведение `is_leaf` на коллизирующих кодах
   (as-is / xfail). Чтобы к пост-замеру был готовый красный тест, а не «надо вспомнить».
2. **Метрика `is_leaf` в пост-замерном diff** — число неверных leaf-rescue из-за коллизии.

Решение чинить/не чинить `is_leaf` — по пост-замеру, отдельной задачей.

## 6. Заметка в TECH_DEBT (не в этот патч)

code-prefix достаточен для главы 6, но на ЭТОЙ смете уже даёт неверную крошку в главах 10–11
(инъекция контекста снизу — 4 узла, измерено; потеря рабочих предков — газон, гл.11). outline-level
восстановление (`outlineLevel` + стек по уровню) на этих узлах даёт корректный результат **здесь и
сейчас** — это не гипотеза «вдруг на других данных», а **уже измеримый** выигрыш на двух главах. Как
примитив outline скорее ПРОЩЕ nearest-preceding-по-коду. Смену структурного сигнала откладываем по
**скоупу** (не из-за неполноценности outline); пост-замер **квантует уже наблюдаемый** промах, а не
проверяет будущую гипотезу.

## 7. Критерии приёмки

- `resolve_ancestor_indices` реализована, возвращает `(индекс, is_fallback)`; инвариант монотонности
  через флаг проверяется тестом.
- Все юнит-фикстуры (§4), включая откат-снизу и property-монотонность, зелёные; сервис-тест
  подтверждает разный `embedding_input` на коллизии.
- Характеризующий тест `is_leaf` присутствует и фиксирует текущее поведение.
- Три точки используют одну функцию; `just test` зелёный.
- Пост-замер `eval_matching` снят (top-1/top-3 дельта + срез 4 узлов отката + метрика `is_leaf`),
  результат записан.
- Заметка в TECH_DEBT добавлена (с честной рамкой про outline по §6).
