## 1. Проблема

Крошка узла сметы (`embedding_input`) собирается из имён предков: для узла с кодом
`6.4.1` предки — коды `6` и `6.4`, их имена клеятся в строку «Имя(6). Имя(6.4). Имя(6.4.1)».
Имя предка по коду берётся из словаря `код → имя`, заполняемого **первым вхождением** кода.

В многоэтапных сметах коды дублируются между этапами с разным смыслом. Пример (глава 6, «Фасады»):

| код | Этап 1 (ЖК) | Этап 2 (БЦ) |
|---|---|---|
| `6.4` | «навесной фасад **типового** этажа» | «навесной фасад **1** этажа» |
| `6.5` | «**модульный** фасад» | «навесной фасад **типового** этажа» |

Словарь «первого вхождения» отдаёт для `6.4` всегда «типового», для `6.5` всегда «модульный».
Поэтому строка `6.4.1` Этапа 2 (реально — «1 этажа») получает предка «типового» → крошка врёт →
вектор уезжает к неверной статье. Баг **тихий** (неверный матч без ошибки), бьёт по всем
многоэтапным сметам.

Корень: предки резолвятся через **глобальный словарь** (теряет позицию), а не по фактическому
положению узла в потоке строк.

## 2. Решение: nearest-preceding (ближайшее предшествующее вхождение)

Имя предка берём не из «первого вхождения за всю смету», а из **ближайшего вхождения того же
кода ВЫШЕ по документу** (макс. `source_index` < узла). Если предшествующего нет — откат к первому
вхождению (= текущее поведение, без потери контекста).

Аналогия: чтобы понять раздел абзаца, человек смотрит на ближайший заголовок над абзацем, а не на
самый первый заголовок документа.

### Почему этого достаточно (а outline-level — оверинжиниринг здесь)

Проверено на golden-смете (вся глава 6, сверка nearest-preceding против outline-восстановления):
- этажный токен в крошке восстанавливается у **всех** коллизионных узлов;
- узлов, где code-prefix и outline дают **разный этаж**: **0**.

Причина: коды **видов работ** внутри главы вкладываются корректно (`6.2.1` под `6.2`, `6.3.1` под
`6.3`). Хаотичны только коды-**обёртки этапов** (этап-заголовок получает код-сосед секций), но они
классифицируются как ORG и **срезаются** из крошки. Поэтому позиционного резолва по коду достаточно.

**Допущение (риск на будущее, см. §6):** достаточность держится на корректном вложении кодов видов
работ. На этой смете держится; на других не проверено.

### Дифференциальный результат на golden-смете

41 узел получает исправленную крошку, **0 регрессий** (ни один не теряет контекст). Бенчмарк 2
(«разделы 1-4») — 0 изменений (нет многоэтапных коллизий).

## 3. Архитектура

### §3.1 Чистая функция в домене

В [domain/classification.py](../../../backend/app/domain/classification.py) (рядом с
`build_embedding_input`):

```python
def resolve_ancestor_indices(codes: Sequence[str]) -> list[list[int]]:
    """codes — коды узлов В ПОРЯДКЕ документа. Для каждой позиции i возвращает
    индексы предков root→parent: каждый код-предок разрешается в БЛИЖАЙШУЮ
    предшествующую позицию с этим кодом (макс. индекс < i); если предшествующей
    нет — первое вхождение этого кода.

    Инвариант: индексы строго возрастают (детектор смешения веток). Предок,
    найденный как ПРЕДШЕСТВУЮЩИЙ, имеет индекс < i. Единственное исключение —
    предок без предшествующего вхождения (код-предка нет выше узла, напр. дочерний
    `10.2.1.1` стоит ВЫШЕ родителя `10.2.1`): откат к первому вхождению, индекс
    может быть ≥ i. Это документированный край, равный текущему поведению (без
    регрессии)."""
```

- Принимает **только упорядоченные коды**, возвращает **индексы** (не имена) → функция чистая,
  без ORM/имён/классов, тривиально тестируется.
- Вызывающий мапит индекс → `(имя, класс)` сам.
- **Инвариант монотонности** — часть контракта: индексы строго возрастают и все `< i`. Защищает от
  патологии «независимый резолв предков собрал цепочку из разных веток» (на golden-смете не
  встречается, но это дешёвый детектор тихой патологии на других данных).

### §3.2 Три точки использования (один источник правды)

Все три заменяют резолв «первого вхождения по коду» на `resolve_ancestor_indices`:

1. **`_classify_nodes`** ([estimate_matching_service.py:154-158](../../../backend/app/services/estimate_matching_service.py#L154))
   — **авторитетный путь**: предки `(имя, класс)` для `build_embedding_input`. Чинит эмбеддинг/матчинг.
2. **`_ancestor_names`** ([estimate_matching_service.py:187](../../../backend/app/services/estimate_matching_service.py#L187))
   — контекст предков для LLM-классификатора (та же коллизия).
3. **`estimate_parser`** ([estimate_parser.py:76-84](../../../backend/app/services/estimate_parser.py#L76))
   — крошка на парсинге. Её перезатирает классификация, но два расходящихся способа строить дерево
   держать нельзя (consistency).

## 4. Тесты (TDD: red → green)

### Юнит на `resolve_ancestor_indices` (чистый, без БД/AI)

Фикстуры из реальных кейсов (имена/смысл — из golden-сметы; gold-метка — оракул истины):

- **Фасад `6.4.1`:** `6.4` ЖК = «навесной типового», БЦ = «навесной 1 этажа» → две строки `6.4.1`
  дают **разных** предков (gold: ЖК→6.4.1 «типового», БЦ→6.3.1 «1 этажа»).
- **Фасад `6.5.1`:** `6.5` ЖК = «модульный», БЦ = «навесной типового» → `6.5.1` БЦ обязан дать
  предка «навесной типового», НЕ «модульный» (gold БЦ→6.4.1). Эта фикстура тестирует именно
  **превосходство nearest-preceding над первым вхождением**.
- **Перегородки `5.1.1`:** заголовок секции во 2-м этапе не повторён → ближайший = первое вхождение
  (контекст сохранён, не orphan).
- **Край (child-before-parent):** код-предок появляется только ПОЗЖЕ узла (реальный `10.2.1.1` выше
  `10.2.1`) → откат к первому вхождению, индекс ≥ i; результат **равен текущему поведению** (без
  регрессии). Тест фиксирует именно это.
- **Property — монотонность:** возвращённые индексы строго возрастают; предки, найденные как
  предшествующие, имеют индекс `< i`. Индекс `≥ i` допустим ТОЛЬКО для краевого отката выше.

### Сервис-тест на `_classify_nodes`

На узлах с коллизирующим кодом `embedding_input` двух строк выходит **разным** (сейчас — байт-в-байт
одинаков). Ловит интеграцию функции в авторитетный слой.

### Пост-замер (разово, не в CI)

`eval_matching` до/после на golden-смете → дельта **top-1 / top-3**. Раз OpenRouter живой — снимаем
честную цифру. **Включить в тот же diff метрику `is_leaf`** (см. §5): срабатывает ли где-то
leaf-rescue неверно из-за коллизии.

## 5. Границы скоупа

**Не трогаем:** org-стрип, порог 0.90, LLM-арбитр, каталог, схему БД.

**Не делаем «корпус»:** иерархический матчинг, смену структурного сигнала и т.п. — отдельный
стратегический разговор после пост-замера распределения промахов.

**`is_leaf` — фикс откладываем, но дисциплинированно.** `is_leaf`
([estimate_matching_service.py:160](../../../backend/app/services/estimate_matching_service.py#L160))
определяется через код-based `parent_codes` и болеет **тем же** механизмом коллизии. В этом патче:

1. **Характеризующий тест сейчас** — фиксирует текущее поведение `is_leaf` на коллизирующих кодах
   (as-is / xfail). Чтобы к пост-замеру был готовый красный тест и доказательство, а не «надо
   вспомнить».
2. **Метрика `is_leaf` в пост-замерном diff** — число неверных leaf-rescue из-за коллизии. Тогда
   «оценить по замеру» — цифра, а не интуиция.

Решение чинить/не чинить `is_leaf` принимается по пост-замеру, отдельной задачей.

## 6. Заметка в TECH_DEBT (не в этот патч)

Достаточность code-prefix держится на корректном вложении кодов видов работ. В этой смете держится;
на других — не проверено. Если пост-замер на новых сметах покажет фасадные промахи — готовый запасной
сигнал — **outline-level** восстановление (на golden-смете дало идентичный результат, значит
совместимо). Строить его сейчас преждевременно.
