# Дизайн: бенчмарк-хранилище gold + оффлайн-метрика качества матчинга

**Дата:** 2026-06-26
**Статус:** черновик на ревью
**Связано:** [2026-06-23-estimate-matching-sp2-design.md](2026-06-23-estimate-matching-sp2-design.md),
[2026-06-25-estimate-org-header-filter-design.md](2026-06-25-estimate-org-header-filter-design.md)
**Следующий под-проект (Спека B):** админ-управление бенчмарками — endpoints (только
admin) + третья вкладка на фронте (список бенчмарков, drill-in, правка ожидаемой
статьи на лету). Зависит от этой спеки; на первую метрику не влияет.

## Декомпозиция

Эта спека — под-проект **A** (хранилище + харнесс). Вместе они дают первую измеримую
пользу (метрику top-1/top-3). Курирующий админ-UI вынесен в **Спеку B** следующим
под-проектом: он зависит от хранилища, но для получения метрики не нужен.

## Проблема

В пайплайне матчинга накоплено много логики: оргфильтр (`status='excluded'`),
чистка крошки от ORG-предков, лексический + LLM-классификатор, порог уверенности
0.90, LLM-арбитр для топ-3. Но **объективной метрики качества нет**. Тюнинг порога,
обогащения крошки и промптов идёт «на глаз»; нельзя ни доказать, что вчерашняя фича
окупилась по точности, ни поймать регрессию (напр. перекос «раздел↔дети»).

Специалист вручную разметил образец сметы до конца: **783 узла** получили эталонную
статью СМР, **26 узлов-группировок** (этапы/корпуса) оставлены без метки. Это
ground-truth, ссылающийся на тот же справочник `template_articles`, что и `Шаблон.xlsx`.
Он даёт инструмент измерения **текущего реализованного пайплайна как есть**.

**Цель v1:** (1) **бенчмарк-хранилище** в БД — gold становится durable приватным
референс-данным (не зависит от локального xlsx); (2) **оффлайн-скрипт**, который
читает gold из БД, прогоняет образец через реальный пайплайн и выдаёт две группы
метрик — качество **классификации** (оргфильтр) и качество **матчинга** (top-1 /
top-3) — плюс пофайловый отчёт расхождений для разбора глазами.

Это фундамент: сам по себе он ничего не улучшает, но превращает все последующие
эксперименты (дедуп крошки, обогащение `embedding_input`, few-shot арбитра, порог) в
измеримый цикл.

## Источник истины: бенчмарк-хранилище в БД

Gold — это **persistent reference data**, родственник `template_articles`: курируемый,
стабильный, не-per-request. Живёт в БД, а не в локальном файле.

**Почему в БД, а не в git-фикстуре:** единственная копия эталона сейчас — xlsx в
`temp/` (gitignore); в git его не закоммитить (структура проекта MR Group), машина
общая. Приватная Neon — уместный durable-дом для таких данных; git — нет.

### Модель данных (новые таблицы)

- `benchmark` — `id`, `name`, `created_at`. Одна размеченная смета = один бенчмарк;
  таблица накапливает несколько смет в будущем.
- `benchmark_node` — `id`, `benchmark_id` (FK→benchmark), `code` (строка раздела,
  напр. `6.3.1`), `name`, `expected_article_code` (**nullable**). **Все** узлы сметы
  (вкл. 26 структурных с `expected_article_code IS NULL`) — иначе дерево/крошка/
  классификация при прогоне будут неполными.
  - `expected_article_code` **без FK** на `template_articles`: справочник правят
    другие департаменты, gold должен пережить правку каталога; рассинхрон (метка
    ссылается на удалённую статью) — это сигнал (корзина `gold_not_in_catalog`), а не
    ошибка целостности.
- Миграция — ручная/автоген Alembic + ORM-модели синхронно (как требует CLAUDE.md).

### Сид из xlsx (разовая админ-операция, CLI)

- Скрипт сидинга: `temp/Смета - образец размеченная до конца.xlsx` → строки `benchmark`
  + `benchmark_node`. Путь — параметр (`--gold <path>`). xlsx нужен **только здесь**;
  в рантайме харнесс читает из БД.
- **Извлечение метки узла:** для строк-узлов (есть `№ раздела`, код целочисленно-
  сегментный) берём колонку «Статья СМР». Код статьи нормализуем тем же правилом, что
  и [template_parser.py:51](../../../backend/app/services/template_parser.py#L51):
  `^\((.*?)\)` → `group(1)` → убрать пробелы → `strip(".")`. Итог: `(6.3.1) Name` → `6.3.1`.
  Пустая ячейка → `expected_article_code = NULL` (структурный узел).
- **Запись в v1 — только CLI-сид.** Правка через admin-API/UI — Спека B; в этой спеке
  бенчмарк иммутабелен после засидки (пере-сид = пересоздать бенчмарк).

### Семантика метки

- `expected_article_code` пуст ⇒ узел **ожидаемо-структурный** (должен быть исключён);
- задан ⇒ узел **ожидаемо-матчируемый** с этим кодом статьи.
- **Гранулярность — узлы** (`EstimateNode`, как решено): меряем ровно то, что
  размечал специалист и что матчит пайплайн. Листовые позиции — вне области.
- **Дубли кодов** (один код раздела в разных этапах, напр. `3.1.4.1` дважды)
  сравниваются **построчно** (per node), а не схлопываются — и пайплайн, и gold
  работают на уровне строки.

## Что считаем «правильным»

Сравнение по **коду статьи** (стабильный ключ), не по имени (у имён косметические
расхождения регистра/точки). Две независимые группы метрик:

### Группа A — классификация (валидация оргфильтра)

Матрица по всем узлам (ожидаемо-структурный = gold пуст; ожидаемо-матчируемый = gold есть):

| | пайплайн исключил | пайплайн оставил |
|---|---|---|
| **gold пуст** (структура) | TN — верный отброс | FP — оставил мусор (дёшево: оператор отбракует) |
| **gold есть** (работа) | **FN — молчаливый пропуск работы (дорого!)** | TP — верно оставил |

Ключевая цифра — **FN** (исключили реальную работу): это самая дорогая ошибка по
асимметрии из оргфильтр-спеки. Идеал: FN = 0, FP мал.

### Группа B — матчинг (только по узлам gold-есть И пайплайн-оставил)

- **top-1 accuracy:** код статьи, выбранной пайплайном (`NodeMatch.article_code`,
  будь то `confident` или `needs_review`), == gold-код.
- **top-3 (retrieval) accuracy:** gold-код присутствует среди топ-3 кандидатов
  (`NodeMatch.candidates[*].code`). Изолирует качество векторного поиска от арбитра.
- **Разбивки:**
  - по статусу: `confident` (порог >0.90) vs `needs_review` (арбитр) vs `no_match`;
  - по сложности: «лёгкие» (имя узла == статья после нормализации) vs «переводы»
    (имя ≠ статья) — на образце это 342 vs 184; интересно видеть accuracy отдельно;
  - bucket **`gold_not_in_catalog`**: gold-код отсутствует в текущем справочнике (специалист
    сослался на статью, которой в `Шаблон.xlsx` уже/ещё нет) — такие узлы top-1/top-3
    в принципе не возьмут; выносим в отдельную корзину, чтобы не занижать accuracy.

**Детерминизм.** Top-3 retrieval детерминирован при фиксированных эмбеддингах. Top-1
для ветки `needs_review` **зависит от LLM-арбитра и может плавать между прогонами** —
поэтому метрики A/retrieval и арбитр-зависимый top-1 в отчёте разделены.

## Подход к прогону: интеграционный (реюз production-пути)

**Выбрано (рекомендация):** прогонять образец через **реальные сервисы против реальной
БД и справочника**, как это делает воркер. Скрипт:

1. Строит реальные адаптеры из `get_settings()` + сессия БД (зеркало
   [deps.py `build_estimate_matching_service`](../../../backend/app/api/deps.py#L175)):
   `OpenRouterEmbedder`, `OpenRouterLLMMatcher`/`Anthropic`, `OpenRouterWorkClassifier`,
   `SqlAlchemyArticleRepository`, `SqlAlchemyEstimateRepository`, `MatchingService`,
   `EstimateMatchingService`.
2. Читает `benchmark_node` из БД и реконструирует **транзиентную смету** напрямую
   через `EstimateRepository` (`code`+`name` → узлы; `parent_code`/глубина выводятся
   из кода), минуя xlsx, S3 и Celery. `embedding_input` пайплайн пересоберёт сам на
   шаге классификации. Сметы в системе и так транзиентны.
3. Вызывает `EstimateMatchingService.match_estimate(estimate_id)` — тот самый
   production-метод (classify → embed → gate → match → status).
4. Читает узлы обратно через репозиторий (статус, выбранная статья, снимок кандидатов),
   сопоставляет с gold, считает метрики A и B.
5. **Удаляет транзиентную смету** (cleanup в `finally`).

*Плюс:* максимальная достоверность — мерим точно тот код, что работает в проде
(включая classify/gate/breadcrumb/арбитра). Дублирования логики в харнессе нет.
*Минус:* пишет транзиентную смету в БД (удаляем после) и делает реальные API-вызовы
(эмбеддинг ~760 узлов + арбитр на ветке `needs_review`). Стоимость разовая и умеренная.

**Отвергнуто — «lean-харнесс»** (переписать оркестрацию в скрипте, вызывая доменные
функции + порты напрямую): меньше мутаций БД, но **дублирует** classify+breadcrumb-логику
из `EstimateMatchingService` → риск дрейфа, харнесс начнёт мерить «почти прод», а не прод.
Достоверность важнее экономии записи в БД.

## Предусловия запуска

- Справочник загружен и **проэмбежен** (`just embed-worker`): харнесс зависит от
  `search_similar` по pgvector. Фейл-фаст: перед прогоном проверить
  `ArticleRepository.matching_readiness()` (`total>0 и pending==0`) и понятно упасть,
  если справочник не готов — иначе gate в `match_estimate` поднимет `DictionaryNotReadyError`.
- Валидный `backend/.env` (ключи OpenRouter/Anthropic, `DATABASE_URL`).

## Размещение и запуск

- **Миграция:** Alembic-ревизия с `benchmark` / `benchmark_node` + ORM-модели синхронно.
- **Сид-скрипт:** `backend/app/scripts/benchmark_seed.py`. Рецепт
  `just benchmark-seed gold="<path>"` → `uv run python -m app.scripts.benchmark_seed --gold ...`.
  Читает xlsx, создаёт бенчмарк + узлы. Разовая админ-операция (CLI, как `create-admin`).
- **Харнесс:** `backend/app/scripts/eval_matching.py` (рядом с `create_admin`, `smoke_import`).
  Рецепт `just eval-matching [benchmark="<name>"]` →
  `uv run python -m app.scripts.eval_matching ...`. Читает gold **из БД**, не из xlsx.
- Флаги харнесса: `--benchmark <name|id>` (какой бенчмарк прогнать; дефолт — если он
  один), `--report <path>` (CSV-детализация; дефолт — gitignored-папка), `--keep` (не
  удалять транзиентную смету, для отладки).
- **Не pytest:** харнесс ходит в реальную БД/AI и стоит денег — это инструмент-прогон,
  а не юнит-тест. Доменные правила извлечения/нормализации gold-кода покрываем
  отдельными юнит-тестами (см. «Тесты»).

## Вывод

1. **Сводка в stdout:** счётчики группы A (TN/FP/FN/TP, явно выделить FN), группы B
   (top-1, top-3 retrieval, разбивки по статусу/сложности, размер `gold_not_in_catalog`).
2. **CSV-детализация** (в `--report`, по умолчанию в gitignored-папку): по строке на
   узел — `code, name, gold_code, kept(bool), status, chosen_code, top3_codes,
   top1_hit, top3_hit, bucket`. Это рабочий материал для разбора расхождений глазами.

## Слои (Clean Architecture)

- **domain:** чистые функции извлечения/нормализации gold-кода (`parse_gold_code`),
  построения матрицы классификации и подсчёта top-1/top-3 — без I/O, тестируемы.
- **infrastructure/db:** `BenchmarkRepository` (создать бенчмарк+узлы при сиде, читать
  узлы при прогоне) + ORM-модели `benchmark`/`benchmark_node`.
- **infrastructure:** чтение xlsx (openpyxl) — адаптер сида.
- **scripts:** `benchmark_seed` (xlsx → БД) и `eval_matching` (БД → транзиентная смета
  → `match_estimate` → чтение результатов → сводка/CSV/cleanup).
- Харнесс реюзает существующие сервисы/порты матчинга — новых портов матчинга не вводим;
  единственная новая абстракция — репозиторий бенчмарков.

## Обработка ошибок

- Справочник не готов / `.env` неполон → понятный фейл-фаст до прогона.
- Падение в середине прогона → транзиентная смета всё равно удаляется (`finally`),
  кроме `--keep`.
- Узлы с транзиентной AI-ошибкой (`status='error'`) пайплайн уже гасит поузлово; в
  отчёте такие узлы попадают в bucket «error» и не засчитываются как промах матчинга.

## Тесты (pytest)

- `parse_gold_code`: `(6.3.1) Name` → `6.3.1`; пустая ячейка → `None`; мусорная
  строка → `None`; trailing-dot/пробелы нормализуются как в `template_parser`.
- Матрица классификации: синтетические пары (gold, kept) → корректные TN/FP/FN/TP.
- Подсчёт top-1/top-3: на синтетических `NodeMatch` + gold — хиты/промахи, bucket
  `gold_not_in_catalog`, разбивка по статусу.
- Сам прогон (сеть/БД) тестом **не покрываем** — это инструмент.

## Вне области (YAGNI / отдельные под-проекты)

- **Спека B — админ-управление бенчмарками:** admin-only endpoints (CRUD над
  `benchmark`/`benchmark_node`, импорт), ролевой гейтинг, **третья вкладка на фронте**
  (список бенчмарков, drill-in, inline-правка ожидаемой статьи на лету с подсветкой
  `gold_not_in_catalog`). Зависит от этой спеки; на первую метрику не влияет.
- **Несколько размеченных смет / агрегат по корпусу смет** — схема `benchmark` это уже
  поддерживает (несколько строк), но харнесс v1 гоняет один бенчмарк за раз.
- **Кэш эмбеддингов** для удешевления повторных прогонов (тексты узлов
  детерминированы) — оптимизация на потом.
- **Метрика на листовых позициях** (протягивание метки узла вниз) — листья наследуют
  метку, ground-truth слабее; добавим, когда понадобится.
- **CI-гейт / порог регрессии** — сначала накопить baseline-числа вручную.
- **Применение результатов** (обогащение `embedding_input`, few-shot арбитра, тюнинг
  порога) — отдельные спеки, которые гоняются *через* эту линейку.
- **Неадъяцентный дедуп крошки** (`MR Base . X . MR Base`) — соседний уже закрыт
  `_collapse_consecutive`; неадъяцентный редок, phase-2.

## Definition of Done

- Миграция `benchmark`/`benchmark_node` применяется (`just migrate`); ORM синхронны.
- `just benchmark-seed gold="temp/Смета - образец размеченная до конца.xlsx"` создаёт
  бенчмарк со **всеми** узлами; пустая «Статья СМР» → `expected_article_code = NULL`.
- `just eval-matching` читает gold **из БД**, реконструирует транзиентную смету,
  прогоняет реальный `match_estimate` и печатает сводку A+B.
- Группа A считает TN/FP/FN/TP; **FN (молчаливый пропуск работы) выделен явно**.
- Группа B считает top-1 и top-3 retrieval с разбивками по статусу и сложности
  (лёгкие/переводы) и корзиной `gold_not_in_catalog`.
- CSV-детализация по узлам пишется в gitignored-путь; данные сметы в репозиторий не
  коммитятся.
- Транзиентная смета удаляется после прогона (кроме `--keep`).
- Фейл-фаст, если справочник не проэмбежен.
- Доменные функции (parse/нормализация/подсчёт) покрыты юнит-тестами; сетевой прогон —
  нет.
