# 2026-06-25 — Фильтрация организационных заголовков в потоке смет (lean v1)

## Что сделано

До эмбеддинга смета классифицирует каждый узел по СМЫСЛУ имени в класс
`WorkClass ∈ {WORK, ORG, UNSURE}` каскадом «дёшево → дорого» (лексика → LLM на смесь). Чистый
организационный заголовок (этап/очередь/корпус/литер/секция/объект + аббревиатуры ЖК/БЦ) обратимо
исключается из эмбеддинга/матчинга — ему ставится терминальный `status='excluded'` (строка не
удаляется, в выгрузке «Статья СМР» пуста). Параллельно из крошки `embedding_input` выживших узлов
выбрасываются ORG-предки (справочник org-free — оргтокены предков тянули косинус правильных пар вниз).

Инвариант дизайна — **асимметрия ошибок**: при любом сомнении класс НЕ `ORG` (ложный ORG = молчаливый
пропуск работы; ложный WORK оператор отбракует на проверке). `UNSURE` трактуется как WORK.

Спек: [docs/superpowers/specs/2026-06-25-estimate-org-header-filter-design.md](../superpowers/specs/2026-06-25-estimate-org-header-filter-design.md).
План: [docs/superpowers/plans/2026-06-25-estimate-org-filter.md](../superpowers/plans/2026-06-25-estimate-org-filter.md).
Ветка `feat/estimate-org-filter` (база `main`). Код-коммиты `b71892f..626d014` (10, по числу задач +
фикс ревью), + 2 docs-коммита в TECH_DEBT (`5b7ab45`, `d2e5263`). PR пока не открыт.

## Архитектура

`api → services → domain ← infrastructure`. Класс `WorkClass` вычисляется **в памяти** в проходе
классификации и **НЕ персистится** — единственный след в БД это `status='excluded'` (колонка `status`
уже `VARCHAR` без CHECK → **миграции не потребовалось**, новой колонки `node_class` нет).

- **domain (чистый, без I/O):** [classification.py](../../backend/app/domain/classification.py) —
  оргтокены (морфологический стем, заякорен на начало слова; литералы ЖК/БЦ), `has_work_word`
  (определён НЕГАТИВНО), `classify_lexical` (каскад), `build_embedding_input` (крошка с выбросом
  ORG-предков, схлопывание повторов, нормализация пробелов). Сущности `WorkClass`, `NodeToClassify`,
  `ClassifiableNode`, `NodeClassification`, `EstimateRowStatus.EXCLUDED`. Порт `WorkTypeClassifier`.
- **infrastructure/ai:** [openrouter_classifier.py](../../backend/app/infrastructure/ai/openrouter_classifier.py)
  — адаптер порта через OpenRouter (см. «Девиация»). Батч имён + строгий JSON + фолбэк в UNSURE.
- **services:** шаг `_classify_nodes` в [estimate_matching_service.py](../../backend/app/services/estimate_matching_service.py)
  — оркестрация каскада сразу после `set_status(RUNNING)`, ДО `_embed_nodes`; пересборка крошки.
- **db:** [estimate_repository.py](../../backend/app/infrastructure/db/estimate_repository.py) —
  bulk-persistence классификации + исключение `excluded` из выборок/счётчиков.

## Бэкенд (9 задач, строгий TDD)

- **Каскад (Tasks 1–3):** `contains_org_token` (стем `этап→этапы/этапов`, инвариант `этап ≠ этаж`
  закрыт тестами; ЖК/БЦ — литералы) → `has_work_word` (аббревиатура 2–4 заглавных = голова; оргтокены
  отсекаются ДО аббрев-правила, иначе ЖК/БЦ ложно сочлись бы головой; «работы/работ» НЕ в стоп-листе) →
  `classify_lexical` (нет оргтокена → WORK; орг+голова → UNSURE; орг без головы → ORG).
- **Чистка крошки (Task 4):** `build_embedding_input` выбрасывает ORG-предков, схлопывает подряд идущие
  повторы, нормализует пробелы (вкл. `\xa0`).
- **Порт + адаптер (Tasks 5–6):** порт `WorkTypeClassifier.classify(items)->list[WorkClass]` (выровнен
  по входу) + фейк; OpenRouter-адаптер с фолбэком «любой сбой/битый JSON/несовпадение длины → весь батч
  UNSURE, НИКОГДА ORG».
- **Persistence (Task 7):** `save_node_classifications` — **bulk, один commit**, с **охраной статуса**:
  пишет только строки в `status IN ('pending','excluded')` (терминальные матч-статусы/ревью
  неприкосновенны); `excluded=True→'excluded'`, `False→'pending'` (обратимый флип на повторном прогоне).
  `fetch_unembedded_nodes` и `nodes_count` исключают `excluded`; `fetch_matchable`/`count_unfinished`/
  `count_node_errors` не трогались — `excluded` вне их наборов статусов (смета не виснет в знаменателе).
- **Оркестрация (Task 8):** собственный класс узла по `n.id` (НЕ по коду — дубли кодов не схлопываются,
  иначе excluded-решение утекло бы между строками с одним кодом → молчаливый пропуск работы);
  представитель по коду — только для имён/классов предков крошки. ORG из ЛЮБОГО источника (лексика ИЛИ
  вердикт LLM) → excluded.
- **Экспорт (Task 9):** характеризующий тест — `_cell_value` для `status='excluded'` → `""` (правок
  кода не требует, фиксирует контракт).

## Девиация от плана: классификатор на OpenRouter, не Anthropic

План предписывал Anthropic-SDK-адаптер. По директиве заказчика классификатор сделан **OpenRouter-only**
— это провайдер проекта по умолчанию (эмбеддер openrouter-only; арбитр дефолтит на openrouter). Адаптер
зеркалит транспорт `OpenRouterLLMMatcher` (httpx `/chat/completions`, Bearer, Referer/Title, `_BodyError`,
`_is_transient`, `retry_transient`); логика промпта/парсинга провайдер-независима. Конфиг:
`classifier_model="anthropic/claude-haiku-4.5"` (слаг **сверен в живом каталоге OpenRouter**),
`classifier_batch_size=40`. DI `get_work_classifier()` ветка openrouter в `api/deps.py`.

## Решения и нюансы

- **Только `status='excluded'`, без `node_class`.** WORK/UNSURE поведенчески не различаются (оба
  матчатся) → отдельный маркер не нужен. Следствие (осознанно): `excluded` недетерминирован между
  прогонами (LLM на смесях может флипнуть), персистентного кэша в v1 нет (phase 2).
- **Оргквалификатор в СОБСТВЕННОМ имени узла не срезается** (phase 2). Поэтому смешанные выжившие
  («Фасадные осветительные приборы 1 Этап ЖК») сохраняют «1 Этап ЖК» в своём имени и в крошке потомков —
  это НЕ дефект, а заявленная граница «ядро / добавка». Срезаются только ORG-предки (отдельные каркасные
  узлы), и это подтверждено на данных (0 утечек именно каркасов).
- **Фолбэк по асимметрии.** Любой сбой классификатора (сеть/битый JSON/несовпадение длины) → весь батч
  UNSURE (keep), никогда ORG. Широкий `except Exception` в адаптере — намеренный контракт спеки.
